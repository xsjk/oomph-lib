4The purpose of this document is to provide a "quick" introduction to the
5fundamental objects in \c oomph-lib. Assuming that you
6- have a basic understanding of the finite element method
7.
8and
9- are reasonably familiar with object-oriented programming in C++,
10.
11the Quick Guide should enable you to solve basic PDE
12problems using \c oomph-lib. You should also consult
13the extensive <A HREF="../../example_code_list/html/index.html">
14list of example codes</A> distributed with the library.

16\section main_how_to How to solve simple PDEs with oomph-lib
17
18 To solve a given PDE with \c oomph-lib, its discretisation
19needs to be specified in a \c Problem object.
20This usually involves the specification of the \c Mesh and
21the element types, followed by the application of the boundary conditions.
22If these steps are performed in the \c Problem constructor,
23the driver code itself can be as simple as:
24\dontinclude one_d_poisson_generic_only.cc
25\skipline main(
26\until }
27
28The amount of work required to represent a given PDE as
29a \c Problem object is problem-dependent:
30- If \c oomph-lib's existing \c Mesh objects and element types
31 are sufficient to discretise the problem, the generation of
32 a fully-functional \c Problem object is straightforward. We shall discuss
33 this case in the section \ref problem.
34- If \c oomph-lib provides the required element type but none of the
35 available meshes are suitable,
36 you will have to write your own \c Mesh object. We shall discuss this in
37 the section \ref mesh. It is also possible to create
38 \c oomph-lib meshes based on the output from third-party (commercial)
39 mesh generators. This is illustrated in
40 <A HREF="../../meshes/third_party_meshes/html/index.html">
41 another example.</A>
42- Finally, the problem might involve a PDE for which \c oomph-lib
43 does not provide any suitable elements. In this case you
44 will need to write your own element. When implementing elements
45 in \c oomph-lib the element formulation is generally sub-divided into
46 (at least) two levels. Geometric elements implement the element geometry
47 (e.g. 2D quad elements, 3D brick elements, etc); these geometric elements
48 act as base classes for specific \c FiniteElements that implement
49 the discretisation of particular PDEs. We discuss the
50 implementation of the two types of elements in two separate sections:
51 - In the section \ref FEequations we describe how to
52 discretise a new system of equations, using existing
53 geometric elements.
54 - In the section \ref FEgeom we discuss how to formulate new geometric
55 elements.
56.
57
58In all cases, we start with a brief description of
59the minimum functionality that must be implemented for the
60various objects. The general principles are then illustrated
61by considering the discretisation of the model problem

115\section problem How to build a Problem
116
117All specific \c Problems should be implemented as
118objects that inherit from the generic \c Problem class. These
119specific \c Problem objects will vary considerably depending
120on the exact details of the problem being solved.
121
122Only a few member functions \e must be implemented for each specific
123\c Problem class:
124- The problem constructor \code SomeSpecificProblem(...) \endcode
125- and the member functions
126 \code Problem::actions_before_newton_solve() \endcode
127 \code Problem::actions_after_newton_solve() \endcode
128.
129\subsection spec The constructor: SomeSpecificProblem(...)
130The Problem constructor \c SomeSpecificProblem(...)
131usually contains the following steps:
132- Construction of the timestepper (if required), e.g.
133 \code Problem::add_time_stepper_pt(new SomeTimeStepper()); \endcode
134
135- Construction of the \c Mesh (which may involve passing a particular
136 element type as a template parameter), e.g.
137 \code Problem::mesh_pt() = new SomeMesh<SomeElement>(...); \endcode
138
139- Specification of the boundary conditions. By default, all nodal
140 values are assumed to be free and their values are initialised to zero.
141 Therefore, we only need to "pin" the values that are prescribed
142 by boundary conditions and assign any nonzero boundary values.
143 For instance, the following instruction "pins" the (single) nodal
144 value at node "0":
145 \code Problem::mesh_pt()->node_pt(0)->pin(0); \endcode
146 A non-zero boundary value can be set by
147 \code Problem::mesh_pt()->node_pt(0)->set_value(0,1.0); \endcode
148 This statement sets the first (= zero-th in C++'s zero-based indexing)
149 nodal value to 1.0.
150
151- Completion of the build of the elements: Elements are typically
152 constructed during the mesh generation process. To allow a
153 generic implementation of this process, mesh constructors
154 use an argument-free constructor to
155 create the elements. Thus, element constructors must not
156 have any arguments. (See the section `\ref new_mesh', below, for an
157 illustration of this process.) If an element requires any global parameters
158 (such as pointers to physical parameters, function pointers
159 to source functions, etc.), these parameters should be passed to the
160 element via suitable access functions, \e after the element has been
161 created by the mesh constructor.
162
163- Assignment of the global and local equation numbers
164 \code Problem::assign_eqn_numbers(); \endcode
165
166
167\subsection actbef The function Problem::actions_before_newton_solve()
168\c oomph-lib treats all problems as non-linear problems and
169employs Newton's method to solve the system of nonlinear algebraic
170equations that results from its spatial (and, in time-dependent
171problems, temporal) discretisation. Within this framework, linear
172problems are special cases for which Newton's method converges
173in one step.
174
175The (pure virtual) member function \c Problem::actions_before_newton_solve()
176should contain everything that \e must be done before a nonlinear
177solve to complete the
178specification of the problem. For example, the function may contain a call to
179update the boundary conditions.
180\code void actions_before_newton_solve() {update_boundary_conditions();}
181\endcode
182
183If desired, finer granularity may be obtained by overloading the
184empty virtual
185functions \c Problem::actions_before_newton_convergence_check(),
186\c Problem::actions_before_newton_step() and/or
187\c Problem::actions_before_implicit_timestep(), which are executed
188before each calculation of the maximum value of the residuals,
189each Newton step or each timestep, respectively. We refer to a
190<A HREF="../../order_of_action_functions/html/index.html">separate
191document</A> for a more detailed discussion of the various
192"action functions" executed by \c oomph-lib's Newton solver.
193
194\subsection actafter The function Problem::actions_after_newton_solve()
195The (pure virtual) member function \c Problem::actions_after_newton_solve()
196should contain
197everything that is to take place after each nonlinear solve. For
198example, the function might
199contain commands to update the values of any "dependent" variables, or
200post-processing commands.
201\code
202void actions_after_newton_solve()
203 {
204 //Update value of the slave variable in namespace GlobalVariables
205 GlobalVariables::slave = node_pt(0)->value(0) + node_pt(0)->value(1);
206
207 //Call the output function
208 ofstream output_file("result.dat");
209 output(output_file);
210 }
211 \endcode
212
213Again, the finer-grained member functions
214\c Problem::actions_after_newton_step() and
215\c Problem::actions_after_implicit_timestep() are provided.

263\section mesh How to build a Mesh
264
265\c oomph-lib provides a large number of fully-functional \c Mesh objects.
266Many of these meshes can easily be adapted to discretise other
267domains, provided the domain has the same topology as the
268original \c Mesh. We shall illustrate this in section \ref distorted_mesh
269below.
270
271
272\subsection new_mesh How to build a completely new Mesh
273
274
275If the domain is too different from any of the existing meshes, a new,
276customised \c Mesh object, \c SpecificMesh, say, must be created.
277The specific mesh should be created as an object that inherits from the
278generic \c Mesh class. To maximise the potential for code-reuse, we
279recommended making the element type a template parameter
280of the \c SpecificMesh class.
281
282\code
283template<class ELEMENT>
284class SpecificMesh : public Mesh
285\endcode
286
287This allows the \c SpecificMesh to be used
288with any \c FiniteElement that is based on the same geometric
289element type (see the section \ref FEgeom for a more detailed discussion of
290geometric elements).
291
292The minimum requirements for a specific mesh object are that it must:
293- Construct the elements and the \c Nodes,
294- Store pointers to the elements in the \c Mesh::Element_pt vector,
295- Store pointers to the \c Nodes in the \c Mesh::Node_pt vector,
296- Set the elements' pointers to their local \c Nodes
297- Set the positions of the \c Nodes.
298.
299
300We suggest that all specific \c Mesh constructors also set up
301an auxiliary lookup scheme that stores information about the mesh
302boundaries. While this is not strictly required, it greatly facilitates
303the application of boundary conditions in complex domains. Firstly,
304the function \c Mesh::set_nboundary(...) must be called to specify the
305number of mesh boundaries. In addition, the generic \c Mesh class
306provides a function \c Mesh::add_boundary_node(i,node_pt),
307which adds the \c Node pointed to by \c node_pt to the \c Mesh's i-th
308boundary and constructs the required
309look-up schemes, including a reverse lookup scheme that informs all
310\c Nodes which mesh boundaries (if any) they are located on.
311(In our simple 1D example there are only
312two boundaries, each consisting of a single Node.)
313Although this step is optional, it \e is required
314during mesh adaptation -- if the reverse lookup scheme has not been
315set up, a warning is issued and the code execution stops when any mesh
316adaptation is attempted. \b Implementation \b detail: \c Nodes that are
317located on mesh boundaries must be defined as \c BoundaryNodes.
318
319The majority of the work required to build a specific \c Mesh
320takes place in its constructor. Typically, the constructor creates
321the elements (of the type specified by the template parameter \c
322ELEMENT) and uses the elements' member function \c
323FiniteElement::construct_node(...) to create the new \c Nodes. The
324equivalent member function \c
325FiniteElement::construct_boundary_node(...)
326is used to create new \c BoundaryNodes.
327
328Here is the complete class definition for a simple, one-dimensional
329(line) mesh which discretises the 1D domain \f$x\in [0,1]\f$ using
330a specified number of equally-spaced elements.
331\dontinclude one_d_poisson_generic_only.cc
332\skipline template<
333\until };
334
335
336To build a \c OneDimMesh with ten elements of type \c SomeElement, say, the
337\c Problem constructor would contain the following
338\code
339Problem::mesh_pt() = new OneDimMesh<SomeElement>(10);
340\endcode

342\subsection distorted_mesh How to adapt an existing Mesh to a different domain shape
343
344Given that mesh generation tends to be a fairly tedious process, one should
345always check whether it is possible to deform an existing \c Mesh
346into a shape that matches the required domain. Provided that the
347new domain has the same topology as the domain represented
348by the original \c Mesh, this can always be done by
349re-positioning the Nodes. Such "deformed" \c Meshes should be implemented
350via inheritance, by deriving the new \c Mesh from an existing one.
351
352Here is an example that illustrates the procedure. Assume we wish to
353solve an equation in the 2D annular domain bounded (in polar
354coordinates) by \f$ r \in [r_{min}, r_{max}] \f$ and \f$ \varphi \in
355[\varphi_{min}, \varphi_{max}] \f$. Inspection of the
356<A HREF="../../meshes/mesh_list/html/index.html"> list of available
357meshes </A> shows that \c oomph-lib does not provide a mesh for
358this geometry. However, there is a \c Mesh object,
359 \c SimpleRectangularMesh, which provides a uniform
360discretisation of a 2D rectangular domain \f$ x\in [0,L_x] \f$
361and \f$ y \in [0,L_y] \f$ with \f$ N_x \times N_y \f$
362quadrilateral elements. Since the topology of the two domains
363is identical, the annular mesh can be implemented in a few
364lines of code:
365
366\code
367//====================================================================
368/// AnnularQuadMesh, derived from SimpleRectangularQuadMesh.
369//====================================================================
370template<class ELEMENT>
371class AnnularQuadMesh : public SimpleRectangularQuadMesh<ELEMENT>
372{
373
374 public:
375
376 /// \short Constructor for angular mesh with n_r x n_phi
377 /// 2D quad elements. Calls constructor for the underlying
378 /// SimpleRectangularQuadMesh; then deforms the mesh so that it fits
379 /// into the annular region bounded by the radii r_min and r_max
380 /// and angles (in degree) of phi_min and phi_max.
381 AnnularQuadMesh(const unsigned& n_r, const unsigned& n_phi,
382 const double& r_min, const double& r_max,
383 const double& phi_min, const double& phi_max) :
384 SimpleRectangularQuadMesh<ELEMENT>(n_r,n_phi,1.0,1.0)
385 {
386
387 // The constructor for the SimpleRectangularQuadMesh has
388 // built the mesh with n_x x n_y = n_r x n_phi elements in the unit
389 // square. Let's reposition the nodal points so that the mesh
390 // gets mapped into the required annular region:
391
392 // Find out how many nodes there are
393 unsigned n_node=nnode();
394
395 // Calculate the value of pi
396 const double pi = 4.0*atan(1.0);
397
398 // Loop over all nodes
399 for (unsigned n=0;n<n_node;n++)
400 {
401 // Pointer to node:
402 Node* nod_pt=node_pt(n);
403
404 // Get the x/y coordinates
405 double x_old=nod_pt->x(0);
406 double y_old=nod_pt->x(1);
407
408 // Map from the old x/y to the new r/phi:
409 double r=r_min+(r_max-r_min)*x_old;
410 double phi=(phi_min+(phi_max-phi_min)*y_old)*pi/180.0;
411
412 // Set new nodal coordinates
413 nod_pt->x(0)=r*cos(phi);
414 nod_pt->x(1)=r*sin(phi);
415 }
416 }
417
418};
419\endcode

440\section finiteelement How to build a FiniteElement
441
442\c oomph-lib provides fully-functional \c FiniteElements for
443the discretisation of a wide range of PDEs. Most of the existing
444elements are constructed in a
445three-level hierarchy, the base being the generic
446\c FiniteElement class. The next level in the hierarchy contains
447geometric elements (e.g. 1D line elements, 2D quad or triangular
448elements, 3D brick elements, etc.). These geometric
449classes form the bases for elements that implement
450the discretisation of particular PDEs. This hierarchy maximises
451the potential for code-reuse, because it allows many different
452specific elements to be derived from the same geometric element.
453
454We shall discuss the implementation of new element types in
455two sections:
456- \ref FEequations
457- \ref FEgeom
458
459<HR>
460
461\subsection FEequations How to implement a new system of equations as a specific FiniteElement
462
463Assume that we wish to design a new element type, \c SpecificElement,
464say, that implements the discretisation of a system of PDEs on an existing
465geometric element. The \c SpecificElement class needs to implement
466the following functions:
467- The function that specifies the number of nodal values
468 required at each of the element's nodes:
469 \code FiniteElement::required_nvalue(n) \endcode
470- The function that computes the element's residual vector:
471 \code FiniteElement::get_residuals(residuals) \endcode
472- The function that computes the element's Jacobian matrix and
473 its residual vector:
474 \code FiniteElement::get_jacobian(residuals,jacobian) \endcode
475- The output function:
476 \code FiniteElement::output(ostream) \endcode
477.
478
479

481member functions and member data to provide
482additional, problem-specific functionality.

615 get_generic_residual_contribution(residuals,jacobian,flag);
616 }
617 \endcode
618
619- The implementation of the specific source function
620 \f$ f(x) \f$ in \c TwoNodePoissonElement::f(...)
621 makes it impossible to use the \c TwoNodePoissonElement to solve
622 the Poisson equation with any other source functions, even though
623 the discretisation of the ODE would otherwise be completely identical.
624 A better implementation would allow the "user" to specify a different source
625 function without having to change the implementation of the
626 element class itself.
627 \n \n
628 Here is a suggestion for an improved implementation, which illustrates the
629 procedure employed in the existing elements in \c oomph-lib:
630 - Introduce a public typedef into the element class to define
631 the required format of the function pointer:
632 \code
633 /// \short Function pointer to source function:
634 /// The source function returns the value of the
635 /// source function at the global coordinate x.
636 typedef double (*PoissonSourceFctPt)(const double& x);
637 \endcode
638 - Add a function pointer to the private data of the \c
639 TwoNodePoissonElement class and initialise it to NULL
640 in the constructor:
641 \code
642 /// \short Function pointer to source function (initialised to
643 /// NULL in the constructor)
644 PoissonSourceFctPt Source_fct_pt;
645 \endcode
646 - Provide an access function for the source function pointer:
647 \code
648 /// \short Access function to pointer to source function
649 PoissonSourceFctPt& source_fct_pt() {return Source_fct_pt;}
650 \endcode
651 - Re-write the \c TwoNodePoissonElement::f(...) function,
652 so that it evaluates the (global) function pointed to by
653 the source function pointer. If possible, provide a default value
654 for the case when the function pointer has not been set:
655 \code
656 /// Evaluate source function at Eulerian position x
657 double f(const double& x) const
658 {
659 //If no source function has been set, return zero
660 //so that the Poisson equation defaults to a Laplace equation.
661 double source=0.0;
662 if(Source_fct_pt!=0)
663 {
664 // Evaluate source function
665 source = (*Source_fct_pt)(x);
666 }
667 return source;
668 }
669 \endcode
670 .
671 The pointer to the source function (typically defined in a suitable
672 namespace in the "user's" driver code) can now be set in the
673 Problem constructor.
674- Generalise the \c TwoNodePoissonElement to a \c
675 TwoNodeSelfAdjointElement that implements the isoparametric
676 discretisation of the self-adjoint ODE
677 \f[
678 \frac{\mbox{d}}{\mbox{d}x}
679 \left( a(x) \frac{\mbox{d}u}{\mbox{d}x} \right) + b(x) u(x)
680 = f(x). \ \ \ \ \ \ \ \ \ (3)
681 \f]
682 Use function pointers to allow the "user" to specify
683 the coefficient functions \f$ a(x), \ b(x) \f$ and \f$ f(x) \f$.

689\subsection FEgeom How to build a new geometric element
690
691Geometric elements define the geometry of a \c FiniteElement. They are
692usually implemented as distinct classes that can then be used to create
693a number of \c FiniteElements each discretising a specific PDE, but
694with the same underlying geometrical representation.
695For this purpose, each geometric \c FiniteElement must
696implement the following functions:
697- The constructor
698 \code SomeGeometricFiniteElement::SomeGeometricFiniteElement() \endcode
699 must perform the following actions:
700 - Set the number of Nodes by calling the protected member function
701 \code FiniteElement::set_n_node(n_node). \endcode
702 - Set the dimension of the element by calling the protected member
703 function \code FiniteElement::set_dimension(dim)\endcode
704 Here the dimension of the element is defined as the number of
705 local coordinates required to parametrise its shape.
706 (Note that this is not necessarily the same as the spatial dimension
707 of the problem -- we can have one-dimensional \c FiniteElements
708 in 3D space, say.)
709 - \b Optional:
710 Set the spatial integration scheme by calling the protected member
711 function \code
712 FiniteElement::set_integration_scheme(&integration_scheme) \endcode
713 - \b Optional:
714 If the spatial dimension (= the number of Eulerian coordinates)
715 required at all Nodes in the element is different from the spatial
716 dimension of the element then set the "nodal dimension"
717 by calling the protected
718 member function \code FiniteElement::set_nodal_dimension(dim)\endcode
719 For example in shell elements, a two-dimensional surface is embedded
720 in a three-dimensional space, so the elemental dimension is two, but
721 the nodal dimension is three.
722 - \b Optional: If the Eulerian coordinate (position) is interpolated using
723 generalised coordinates, e.g. vector-valued shape functions, or
724 Hermite-type interpolation, set the number of generalised
725 coordinates or "position types" by using the protected member function
726 \code FiniteElement::set_n_nodal_position_type(n_position_type)\endcode
727
728- The function that computes the element's (geometric) shape functions
729 at specified values of the local coordinates. These shape
730 functions are primarily used to implement the mapping between the
731 element's local coordinates and the global (Eulerian) coordinates.
732 The geometric shape functions \e can be (and often are) also used in
733 specific \c FiniteElements to interpolate the unknown
734 function(s) between the nodal values.
735 \code FiniteElement::shape(s,psi) \endcode
736
737It is usually necessary to implement the following additional functions:
738- The function that computes the derivatives of the element's shape
739 functions with respect to the local coordinates,
740 \code FiniteElement::dshape_local(s,psi,dpsids) \endcode
741- The function that specifies the number of Nodes along the
742 element's 1D "edges"
743 \code FiniteElement::nnode_1d() \endcode
744.
745Further member functions and member data may be defined as and when required.
746For instance, all specific \c FiniteElements must store a pointer to an
747(instantiated) spatial integration scheme. It is good practice to
748provide a default integration scheme for each geometric element and
749to ensure its instantiation by making it
750a static data member of the geometric element class. This allows the
751constructor of the geometric element to set the pointer to the default
752integration scheme. If the default is not appropriate for a
753specific derived \c FiniteElement, the default assignment
754can be over-written in the constructor of the derived class.
755
756As a concrete example, we consider the implementation of the

881\subsection pre_compute_psi Using pre-computed shape functions and the StorableShapeElement<ELEMENT> class
882
883 The computation of the element's residual vector and the Jacobian
884matrix requires the evaluation of the element's shape functions,
885\f$ \psi_j \f$, and their derivatives with respect to the global
886coordinate, \f$ \mbox{d}\psi_j/\mbox{d}x \f$, at the
887element's integration points. In our simple example class,
888\c TwoNodePoissonElement, we (re-)compute these functions
889"on the fly", using the function
890\code
891FiniteElement::dshape_eulerian(s,psi,dpsidx)
892\endcode
893where the local coordinate of the integration point is passed
894in the (one-dimensional) vector \c s.
895
896 The re-computation is wasteful because:
897- The values of the shape functions and their derivatives, evaluated
898 at the integration point, are independent of the value of the
899 \c Sign variable. Their re-computation during the second
900 solve could therefore be avoided if we stored their values
901 in a suitable container.
902- The values of the shape functions at the integration points
903 are identical for all elements. Therefore we only need to provide
904 storage for the shape function values in a single, representative
905 element. The other elements can access the values via pointers.
906.
907To circumvent these potential inefficiencies, \c oomph-lib
908provides alternative interfaces for various functions that compute
909shape functions and their derivatives:
910- Calls to the function
911 \code
912 FiniteElement::shape(s,psi)
913 \endcode
914 which computes the shape function at a given local coordinate, \c s,
915 can be replaced by
916 \code
917 FiniteElement::shape_at_knot(int_point,psi)
918 \endcode
919 where the \c unsigned argument \c int_point identifies
920 the integration point (as specified by the
921 element's spatial integration scheme).
922 In "standard" \c FiniteElements the function
923 \c FiniteElement::shape_at_knot(...) simply determines the position
924 of the integration point and calls \c FiniteElement::shape(...) and
925 so the shape functions are still computed "on the fly".
926 The templated class \c StorableShapeElement<ELEMENT>, however,
927 may be used to
928 upgrade any class derived from \c FiniteElement into a class that \b
929 can store the values of the shape functions and their derivatives at
930 the integration points. The function \c shape_at_knot(...) is
931 overloaded so that when called for the first
932 time, it computes the values of the shape function
933 at all integration points and stores them for future reference.
934 In subsequent calls, the function returns the stored values, rather
935 than re-computing them.
936- There are equivalent alternatives for the functions that
937 compute the shape functions and their derivatives with respect
938 to the local coordinates:
939 \code
940 FiniteElement::dshape(s,psi,dpsids)
941 \endcode
942 can be replaced by
943 \code
944 FiniteElement::dshape_at_knot(int_point,psi,dpsids)
945 \endcode
946 while
947 \code
948 FiniteElement::d2shape(s,psi,dpsids,d2psids)
949 \endcode
950 can be replaced by
951 \code
952 FiniteElement::d2shape_at_knot(int_point,psi,dpsids,d2psids)
953 \endcode
954.
955These functions are all overloaded in the \c
956StorableShapeElement<ELEMENT> class and by default the
957overloaded functions store the pre-computed values of the
958shape functions and their derivatives locally within each element.
959This implementation ensures data locality and should increase
960the speed of access to the stored values. However, it
961can also create significant storage overheads. \c oomph-lib
962therefore provides the function
963\code
964StorableShapeElement<ELEMENT>::set_shape_local_stored_from_element(...)
965\endcode
966which frees up the storage in the element and replaces it by
967pointer-based access to the values stored in another element.
968Typically, all elements in a mesh are of the same type and use
969the same spatial integration scheme and so the values of the shape
970functions need only be stored in one element.
971
972Derivatives of the element's shape functions with respect to
973the global Eulerian coordinates are generally
974computed by
975\code
976FiniteElement::dshape_eulerian(s,psi,dpsidx)
977\endcode
978and
979\code
980FiniteElement::d2shape_eulerian(s,psi,dpsidx,d2psidx)
981\endcode
982These functions also return the Jacobian of the mapping between
983the local and global coordinates -- the Jacobian can also be
984computed independently from
985\code
986FiniteElement::J_eulerian(s)
987\endcode
988The derivatives of the shape functions with respect the
989global coordinates depend on the nodal positions, and it is not safe
990to assume that they will remain constant. For instance
991in moving boundary problems, the position of the nodes is
992determined as part of the solution. By default, the functions
993\code
994StorableShapeElement<ELEMENT>::dshape_eulerian_at_knot(int_point,psi,dpsidx)
995\endcode
996\code
997StorableShapeElement<ELEMENT>::d2shape_eulerian_at_knot(int_point,psi,dpsidx,d2psidx)
998\endcode
999and
1000\code
1001StorableShapeElement<ELEMENT>::J_eulerian_at_knot(int_point)
1002\endcode
1003re-compute the derivatives at the specified integration
1004point "on the fly". If the "user" is confident
1005that in his/her problem, the nodal positions will not change, the values
1006can be pre-computed by calling
1007\code
1008StorableShapeElement<ELEMENT>::pre_compute_dshape_eulerian_at_knots()
1009\endcode
1010\code
1011StorableShapeElement<ELEMENT>::pre_compute_d2shape_eulerian_at_knots()
1012\endcode
1013\code
1014StorableShapeElement<ELEMENT>::pre_compute_J_eulerian_at_knots()
1015\endcode
1016Once these functions have been called, any subsequent calls to
1017the \c *_eulerian_at_knot(...) functions return the stored values.
1018To revert to the case in which the derivatives are re-computed
1019"on the fly", the storage for the derivatives must be deleted by
1020calling
1021\code
1022StorableShapeElement<ELEMENT>::delete_dshape_eulerian_stored()
1023\endcode
1024
1025\b Notes:
1026- In the (unlikely!) case that an element's spatial integration
1027 scheme is changed during the code execution, all stored
1028 values are automatically re-computed.
1029- The class
1030 \c StorableShapeSolidElement<ELEMENT> provides equivalent overloaded
1031 functions for derivatives
1032 with respect to the element's Lagrangian coordinates that are used in solid
1033 mechanics problems.

1038\subsection equation_classes Further sub-division of specific element classes -- equation classes.
1039
1040We suggested that any newly developed elements should be constructed
1041in a three-level hierarchy, \c FiniteElement -- Geometric Element --
1042Specific \c FiniteElement. Most finite elements in \c oomph-lib
1043incorporate an additional intermediate "equation class" which implements the
1044computation of the element residual vector and the element Jacobian
1045matrix in terms of abstract shape and test functions, defined
1046as pure virtual functions in the "equation class". This makes it
1047easy to change the specific element formulation, without having to
1048re-implement the weak form of the governing equation.
1049
1050Note that different element types may store the same physical variable at
1051different locations. For example, the pressure in the Navier--Stokes
1052equations may be stored as internal \c Data (discontinuous) or nodal
1053\c Data
1054(continuous). Particular equation classes may require
1055internal numbering schemes that store the appropriate
1056local equation numbers for each physical variable.
1057These schemes must be assembled for each specific element in
1058the function
1059\c GeneralisedElement::assign_additional_local_eqn_numbers(), which is called
1060from within \c Problem::assign_eqn_numbers().
1061
1062 As an example, consider the weak form of the 2D advection diffusion
1063equation
1064\f[
1065\int \left(\mbox{Pe} \ \sum_{i=1}^2 w_i \frac{\partial u}{\partial x_i} \psi^{(test)}_l +
1066\sum_{i=1}^2 \frac{\partial u}{\partial x_i}
1067 \frac{\partial \psi^{(test)}_l}{\partial x_i} \right) dA
1068 = 0,
1069\f]
1070where the Peclet number, \f$ Pe \f$ , and the "wind" \f$w_i \ (i=1,2)\f$
1071are given. We expand the unknown function \f$u(x_1,x_2)\f$ in terms of
1072the (global) basis functions \f$ \psi^{(basis)}_k(x_1,x_2), \ (k=1,...,N)\f$,
1073\f[
1074u(x_1,x_2) = \sum_{k=1}^N U_k \ \psi^{(basis)}_k(x_1,x_2),
1075\f]
1076where \f$N\f$ is the total number of nodes in the mesh.
1077The mapping between the element's local and global coordinates
1078is represented in terms of the local shape functions
1079\f$\psi^{(shape)}_j(s_1,s_2)\f$ as
1080\f[
1081x_i = \sum_{j=1}^n X_{ij} \ \psi^{(shape)}_j(s_1,s_2)
1082\f]
1083where \f$ n \f$ is the number of nodes in the element.
1084
1085 The following sketch illustrates how this discretisation
1086is implemented in \c oomph-lib's \c QAdvectionDiffusionElement --
1087an isoparametric, quadrilateral element, based on the Galerkin
1088discretisation of the weak form with
1089\f$ \psi^{(shape)}_j = \psi^{(test)}_j = \psi^{(basis)}_j \f$
1090
1091\image html advection_diffusion_element_inheritance.gif "Typical inheritance diagram for oomph-lib elements "
1092\image latex advection_diffusion_element_inheritance.eps "Typical inheritance diagram for oomph-lib elements " width=0.75\textwidth
1093
1094At large Peclet number, the Galerkin discretisation of the advection
1095diffusion equation is well-known to produce spurious "wiggles" in the
1096solution. These can be suppressed by SUPG stabilisation which
1097employs test functions, \f$ \psi_j^{(test)}
1098\f$, that differ from the basis function, \f$ \psi_j^{(basis)} \f$,
1099\f[
1100\psi_j^{(test)} = \psi_j^{(basis)} +
1101\tau^{(SUPG)} \sum_{i=1}^2 w_i \frac{\partial
1102\psi_j^{(basis)}}{\partial x_i},
1103\f]
1104where \f$ \tau^{(SUPG)} \f$ is a stabilisation parameter.
1105This can be implemented with a trivial change to the
1106\c QAdvectionDiffusionElement class -- the
1107\c QSUPGAdvectionDiffusionElement simply provides a different
1108implementation of the test functions.

1113\subsection paranoia Implement extensive (but optional) self tests in all classes
1114
1115We have already illustrated the use of \c oomph-lib's generic \c self_test()
1116functions. The top-level \c Problem::self_test() function
1117performs a systematic test of all fundamental objects
1118involved in a specific problem and can be used to diagnose
1119any problems. It is good practise to implement
1120further \c self_test() functions in any newly developed classes.
1121The generic \c self_test() functions are defined to be virtual
1122functions in \c oomph-lib's fundamental objects and can
1123be overloaded. Obviously, the \c self_test() function in a
1124specific derived object should still call the \c self_test()
1125function of the underlying fundamental object. The
1126\c TwoNodePoissonElement::self_test() function, listed in
1127in the section \ref FEequations illustrates the procedure.
1128
1129 While frequent sanity checks are helpful during code-development,
1130they can introduce significant overheads into the
1131code execution. \c oomph-lib therefore provides a compiler flag PARANOID,
1132which allows the execution of sanity checks to be switched on
1133or off. When developing new classes, sanity checks should
1134be implemented to catch any potential problems, but the relevant
1135code should be surrounded by <CODE>ifdef/endif</CODE> statements to allow
1136the tests to be disabled. Here is an example:
1137\code
1138
1139 #include <typeinfo>
1140
1141 [...]
1142
1143 // Recast to a different pointer type
1144 SomeOtherObject* other_pt=dynamic_cast<SomeOtherObject*>(some_pt);
1145
1146 #ifdef PARANOID
1147 if (other_pt==0)
1148 {
1149 std::ostringstream error_stream;
1150 error_stream << "Failed to cast some_pt to SomeOtherObject* "
1151 << std::endl;
1152 error_stream << "The pointer some_pt points to an object of type: " <<
1153 << typeid(some_pt).name() << std::endl;
1154 throw OomphLibError(error_stream.str(),
1155 OOMPH_CURRENT_FUNCTION,
1156 OOMPH_EXCEPTION_LOCATION);
1157 }
1158 #endif
1159
1160[...]
1161
1162\endcode
1163Note that we throw a specific \c OomphLibError object after catching
1164the error -- this allows the provision of more explicit (and hopefully more
1165meaningful) error messages.
1166
1167 Many access functions that provide indexed access to
1168a private container, do, in fact, access a private STL vector.
1169Explicit range checking for these (frequent!) cases can be avoided
1170by changing the container to \c oomph-lib's \c Vector class. The
1171\c Vector class
1172performs automatic range checking, if \c oomph-lib's \c generic library is compiled
1173with the \c RANGE_CHECKING flag set (i.e. if
1174\c -DRANGE_CHECKING is specified as a compilation flag for the
1175C++ compiler). For access functions that do not use the \c Vector
1176class you should implement your own range checks using the
1177RANGE_CHECKING compiler flag.

1182

1184that a nodal value is
1185pinned, i.e. prescribed by boundary conditions. This convention is used

1200\subsection multi_physics Facilitating the use of specific elements in multi-physics problems
1201
1202As the name of the library suggests, \c oomph-lib's existing
1203"single-physics" elements are (and any newly designed ones should be)
1204designed so that they can easily be used in multi-physics problems.
1205We anticipate two types of multi-physics interactions:
1206- The solution of one system of equations affects the geometry of
1207 the domain in which another equation is solved -- this
1208 occurs, for instance, in fluid-structure interaction problems.
1209- Two physical phenomena occur in the same domain and
1210 interact with each other, so that forcing terms in one
1211 equation depend on the solution of another -- this occurs, for
1212 instance, in thermo-elasticity problems where the temperature
1213 distribution (determined by the solution of an unsteady heat
1214 equation, say) affects the "growth term" in the equations of
1215 elasticity.
1216.
1217
1218 Multi-physics elements for such problems are usually implemented by combining
1219two (or more) single-physics elements via multiple inheritance. The combined,
1220multi-physics element will obviously have a larger number of degrees
1221of freedom than the constituent single-physics elements. The combined element
1222typically calls the \c get_residuals(...) function of the two
1223constituent elements and concatenates their residual vectors.
1224When computing the Jacobian matrix, the "single physics" elements
1225provide the diagonal blocks for the Jacobian matrix of the
1226multi-physics element, while the off-diagonal interaction blocks must
1227be computed separately. The details of the implementation vary
1228from problem to problem. However, any single-physics element
1229must satisfy the following requirements if it is to be used
1230as a base class for derived multi-physics elements:
1231- \ref dont_wipe
1232- \ref virtual
1233- \ref index_fcts
1234- \ref mesh_movement
1235.

1240\subsubsection dont_wipe Rule 1: Elements must only initialise their own entries in the element residual vector and the element Jacobian matrix
1241
1242The computation of the element residual vector and
1243the element Jacobian matrix typically involves a loop over the
1244element's integration points, in which each integration point adds
1245a further contribution to the entries. In multi-physics problems
1246it is crucial that the initialisation of the residual vector
1247and the Jacobian matrix only sets the element's "own" entries to zero.
1248
1249
1250To illustrate this point, consider what would happen if we used
1251the \c TwoNodePoissonElement::get_jacobian(...) function,
1252discussed in the section \ref jac, in a derived multi-physics element,
1253which combines the \c TwoNodePoissonElement with another
1254element, \c TwoNodeSomeOtherEquationElement, say.
1255Assume that we implement the function \c get_jacobian(...) of the
1256combined element so that it first calls the function \c
1257TwoNodeSomeOtherEquationElement::get_jacobian(...) to determine
1258the first diagonal block in the combined Jacobian matrix.
1259When we call \c TwoNodePoissonElement::get_jacobian(...)
1260to compute the entries in the second diagonal block,
1261the initialisation loop
1262
1263\code
1264//Find the number of degrees of freedom (unpinned values) in the element
1265 unsigned n_dof = ndof();
1266//Initialise all entries of the Jacobian matrix to zero
1267for(unsigned i=0;i<n_dof;i++)
1268 {
1269 for(unsigned j=0;j<n_dof;j++) {jacobian(i,j) = 0.0;}
1270 }
1271\endcode
1272
1273would initialise the \e entire Jacobian matrix, thus wiping out
1274the entries that were already computed by
1275\c TwoNodeSomeOtherEquationElement::get_jacobian(...).
1276
1277 The strategy used in \c oomph-lib to permit the easy combination of
1278 elements is to use the two protected member functions of the
1279 \c GeneralisedElement class:
1280\code
1281virtual void GeneralisedElement::fill_in_contribution_to_residuals(Vector<double>&residuals);
1282\endcode
1283
1284and
1285
1286\code
1287virtual void GeneralisedElement::fill_in_contribution_to_jacobian(Vector<double> &residuals,
1288 DenseMatrix<double> &jacobian);
1289\endcode
1290
1291 These functions DO NOT initialise the entries of the residuals vector
1292 or the Jacobian matrix. Instead, the functions merely add the
1293 appropriate contributions to the vector and the matrix entries. The
1294 default version of
1295 the \c get_residuals() and \c get_jacobian() functions, defined in
1296 \c GeneralisedElement, are simple
1297 wrappers that initialise the residuals and Jacobian to zero and then
1298 call the appropriate \c fill_in_contribution... function.
1299\code
1300 virtual void GeneralisedElement::get_residuals(Vector<double> &residuals)
1301 {
1302 //Zero the residuals vector
1303 residuals.initialise(0.0);
1304
1305 //Add the elemental contribution to the residuals vector
1306 fill_in_contribution_to_residuals(residuals);
1307 }
1308
1309 virtual void GeneralisedElement::get_jacobian(Vector<double> &residuals,
1310 DenseMatrix<double> &jacobian)
1311 {
1312 //Zero the residuals vector
1313 residuals.initialise(0.0);
1314
1315 //Zero the Jacobian matrix
1316 jacobian.initialise(0.0);
1317
1318 //Add the elemental contribution to the residuals vector and Jacobian
1319 fill_in_contribution_to_jacobian(residuals,jacobian);
1320 }
1321\endcode
1322
1323 The \c get_residuals function, for example, can thus be overloaded
1324 in a multi-physics element, as follows:
1325\code
1326
1327 /// Multi-physics element, created by multiple inheritance
1328 class SomeMultiPhysicsElement : public virtual TwoNodePoissonElement,
1329 public virtual TwoNodeSomeOtherEquationElement
1330 {
1331
1332 [...]
1333
1334 /// Residual vector of the combined element is made from the entries
1335 /// of the constituent single-physics elements
1336 virtual void get_residuals(Vector<double> &residuals)
1337 {
1338 //Zero the residuals vector
1339 residuals.initialise(0.0);
1340
1341 //Add the first elemental contribution to the residuals vector
1342 TwoNodePoissonElement::fill_in_contribution_to_residuals(residuals);
1343
1344 //Add the second elemental contribution to the residuals vector
1345 TwoNodeSomeOtherEquationElement::fill_in_contribution_to_residuals(residuals);
1346 }
1347
1348
1349 [...]
1350
1351 };
1352
1353\endcode
1354
1355 It is, therefore, recommended that authors of "single-physics"
1356 elements, overload \c fill_in_contribution_to_residuals(...) and
1357 \c fill_in_contribution_to_jacobian(...), rather than \c
1358 get_residuals() and \c get_jacobian(...), respectively. A further
1359 advantage of the implementation is that the author need not
1360 worry about initialisation of the residuals vector or the Jacobian
1361 matrix when using the \c "fill_in_" rather than the \c "get_" functions.

1363\subsubsection virtual Rule 2: Forcing functions, etc. should be implemented as uniquely-named virtual functions
1364
1365To allow for an interaction between multiple equations, any
1366forcing functions (such as the source function in the Poisson
1367equation) should be implemented as virtual functions. This allows
1368them to be overloaded in derived, multi-physics elements where the
1369forcing function in one equation might depend on the unknowns in
1370another one.
1371
1372Furthermore, to avoid clashes of function names that may occur when two
1373single-physics elements are combined, member functions that can
1374be expected to have counterparts in the context of other equations should
1375be given suitable modifiers. For instance,
1376\code
1377AdvectionDiffusionEquation::source_fct_adv_diff(...)
1378\endcode
1379is a better name for a member function that returns the source function
1380in the advection-diffusion equations than
1381\code
1382AdvectionDiffusionEquation::source_fct(...)
1383\endcode
1384since many other equations are likely to have source functions, too.
1385It is obviously impossible to completely avoid such clashes but
1386this strategy makes them less likely to occur. We are fairly confident
1387that the relevant member functions of all existing elements
1388in \c oomph-lib have been given suitable modifiers to avoid
1389clashes of this type. Therefore, you should be able to combine any
1390element with any other element in the library. If you find a counter-example
1391 <A HREF="../../contact/html/index.html">let us know</A>, and
1392we will rectify this in the next release. In the meantime
1393exploit the fact that \c oomph-lib is an open source library; you can
1394change anything you want!

1396\subsubsection index_fcts Rule 3: Nodal values should only be accessed indirectly via index functions that can be overloaded in derived multi-physics elements
1397
1398When implementing the single-physics \c TwoNodePoissonElement,
1399discussed above, we provided an access function \c u(n)
1400that returns the (single) nodal value, stored at the element's
1401\c n -th \c Node. The function was implemented as a simple wrapper
1402that followed the pointer to the element's \c n -th \c Node and
1403returned the zero-th nodal value stored at that \c Node:
1404\code
1405double TwoNodePoissonElement::u(const unsigned &n)
1406 {
1407 return node_pt(n)->value(0);
1408 }
1409\endcode
1410
1411In a single-physics context, this implementation is perfectly
1412acceptable since we know \e a \e priori that in a scalar problem
1413each \c Node only stores a single value.
1414
1415The same logic suggests that a \c TwoNodeAdvectionDiffusionElement
1416would have a member function
1417\code
1418double TwoNodeAdvectionDiffusionElement::u(const unsigned &n)
1419 {
1420 return node_pt(n)->value(0);
1421 }
1422\endcode
1423
1424However, merging these two elements via multiple inheritance
1425creates two problems. First we have clash of names which could
1426have been avoided by following rule 2 above and calling the two functions
1427\c TwoNodePoissonElement::u_poisson(...) and
1428\c TwoNodeAdvectionDiffusionElement::u_adv_diff(...), say.
1429More serious is that both elements regard the zero-th nodal value
1430as "their own". This can (and should!) be avoided
1431by the following re-implementation:
1432
1433\code
1434double TwoNodePoissonElement::u(const unsigned &n)
1435 {
1436 return node_pt(n)->value(u_index_poisson());
1437 }
1438\endcode
1439
1440where the virtual function \c u_index_poisson() provides the
1441default index at which the Poisson nodal values are stored in a
1442single-physics context:
1443
1444\code
1445virtual unsigned TwoNodePoissonElement::u_index_poisson()
1446 {
1447 // By default (i.e. in a single-physics context) the Poisson
1448 // value is stored at the zero-th nodal value:
1449 return 0;
1450 }
1451\endcode
1452
1453The advection-diffusion element can be modified in the same way:
1454
1455\code
1456double TwoNodeAdvectionDiffusionElement::u(const unsigned &n)
1457 {
1458 return node_pt(n)->value(u_index_adv_diff());
1459 }
1460\endcode
1461
1462with
1463
1464\code
1465virtual unsigned TwoNodeAdvectionDiffusionElement::u_index_adv_diff()
1466 {
1467 // By default (i.e. in a single-physics context) the advection-diffusion
1468 // value is stored at the zero-th nodal value:
1469 return 0;
1470 }
1471\endcode
1472
1473In a combined multi-physics problem, we can now merge the two elements by
1474multiple inheritance,
1475
1476\code
1477class TwoNodeAdvDiffAndPoissonElement :
1478 public virtual TwoNodePoissonElement,
1479 public virtual TwoNodeAdvectionDiffusionElement
1480\endcode
1481
1482Name clashes are already avoided, so we only have to overwrite the two index
1483functions to indicate which (scalar) value is stored as which
1484nodal value:
1485
1486\code
1487unsigned TwoNodeAdvDiffAndPoissonElement::u_index_adv_diff()
1488 {
1489 // In the combined multi-physics element we continue to store the
1490 // advection-diffusion value at the zero-th nodal value
1491 // [as a result it is not actually necessary to re-implement this
1492 // function!]
1493 return 0;
1494 }
1495\endcode
1496
1497and
1498
1499\code
1500unsigned TwoNodeAdvDiffAndPoissonElement::u_index_poisson()
1501 {
1502 // In the combined multi-physics element we store the Poisson
1503 // value at the first nodal value:
1504 return 1;
1505 }
1506\endcode
1507
1508Specific examples that illustrate the creation of (non-trivial)
1509multi-physics elements by multiple inheritance are provided
1510in the <A HREF="../../example_code_list/html/index.html#multi">
1511list of example codes.</A>

1515\subsubsection mesh_movement Rule 4: Time derivatives should be implemented in the ALE formulation to allow for moving meshes.
1516
1517In moving mesh problems, the time-derivative of a nodal value
1518(approximated by the Node's \c TimeStepper) is not the same as
1519the Eulerian time-derivative \f$ \partial/ \partial t \f$
1520that "usually" occurs in the governing equation.
1521The former represents the rate of change when following the
1522moving node; the latter is the rate of change evaluated at a fixed
1523Eulerian position. The two time-derivatives are related to each other
1524via the ALE relation
1525\f[
1526\left. \frac{\partial (.)}{\partial t} \right|_{Eulerian}
1527= \left. \frac{\partial (.) }{\partial t} \right|_{at \ node} - \
1528\sum_{i=1}^D v_i^{(Mesh)} \ \frac{\partial (.) }{ \partial x_i}
1529\f]
1530where \f$ D \f$ is the spatial dimension of the problem and
1531\f$ v_i^{(Mesh)} \ (i=1,...,D) \f$ is the velocity of the node.
1532Any Eulerian time-derivatives in the governing equation should
1533be implemented in the above form to ensure that the
1534element remains functional in moving mesh problems.
1535See the <A HREF="../../unsteady_heat/two_d_unsteady_heat_ALE/html/index.html">
1536example describing the solution of the unsteady heat equation in a
1537moving domain</A> for further details.
