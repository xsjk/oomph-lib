3\mainpage Existing (structured) meshes
4
5This document provides a brief description of the various
6structured meshes that are distributed with the library.
7Most of these meshes were developed for specific example codes but we expect
8them to be useful in other problems too. Many of the meshes
9exist in many different variants, usually constructed
10by multiple inheritance. When meshes are relatively trivial
11variations of each other, e.g. a basic mesh and its refineable equivalent,
12we only list the mesh once. The detailed documentation for the mesh
13(obtained by following the link) contains the full inheritance
14diagram, showing the mesh's own base classes and any meshes that are derived
15from it. For each mesh, we provide a link to a fully-documented
16example problem that illustrates its use.
17
18
19 We stress that the <B>key feature of any given mesh</B> is
20<B>its topology</B>, rather than the specific shape for which it
21was originally developed. For instance, \c oomph-lib does not
22provide a mesh for the discretisation of an annular
23domain with quadrilateral elements. However, such a mesh is
24trivial to construct by deriving it from, e.g., the
25\c SimpleRectangularQuadMesh (a mesh that discretises a
26rectangular domain), and then adjusting its nodal positions.
27Consult the example in the
28<A HREF="../../../quick_guide/html/index.html#distorted_mesh">
29(Not-So-)Quick-Guide</A> for details.

34\section conventions Reminder: General conventions to facilitate the re-use of meshes
35Since mesh generation tends to be most tedious part of any numerical
36simulation, \c oomph-lib's overall data structure, described in
37detail <A HREF="../../../the_data_structure/html/index.html">elsewhere,
38</A> aims to facilitate the re-use of meshes in many different
39applications. For this purpose most specific \c FiniteElements in \c oomph-lib
40are derived by multiple inheritance, combining a "geometric" \c FiniteElement
41(e.g. a line/quadrilateral/brick-shaped element from the
42\c QElement<DIM,NNODE_1D> family) with an equations class
43(such as \c PoissonEquations<DIM>) that implements the
44weak form of a specific PDE. For instance, \c oomph-lib's
45quadrilateral nine-node Poisson element, \c QPoissonElement<2,3>,
46is derived from the \c PoissonEquations<2> equations class and the
47\c QElement<2,3> geometric \c FiniteElement.
48
49The mesh generation process
50is mainly concerned with the geometric properties of the mesh's
51constituent \c FiniteElements (their topology, number of
52nodes, etc.) which are defined by the geometric
53\c FiniteElement. This makes it possible to use a mesh that
54was originally developed for the solution of
55a Poisson equation with a \c QPoissonElement<2,NNODE_1D>, say,
56for the solution of an advection-diffusion problem
57with a \c QAdvctionDiffusionElement<2,NNODE_1D> since both
58elements are derived from the same geometric \c FiniteElement.
59Only two aspects of the mesh generation process require
60information that is not provided by the geometric \c FiniteElement:
61- <B>The number of values to be stored at the various nodes in the mesh:</B>
62 For instance, if a mesh that is designed for quadrilateral elements
63 from the \c QElement<2,NNODE_1D> family of geometric elements is used
64 to solve a (scalar) Poisson equation with nine-noded elements,
65 each \c Node has to store a single value. However, if the same mesh is
66 used for the solution of the 2D Navier-Stokes equations with a
67 nine-node quadrilateral elements of type \c QTaylorHoodElement<2>,
68 nodes located
69 at the elements' four vertices have to store three values (two
70 velocity components and one pressure) whereas nodes located
71 at the elements' interior and on their edges only have to store
72 two values (the two velocity components). \n\n
73- <B>The number of "history values" required by the timestepping
74 procedure (if any):</B> For instance, if the mesh is used for the
75 solution of a Poisson problem, no "history" values are required. If the same
76 mesh is used for the solution of an unsteady heat problem, the
77 number of history values required is determined by the \c
78 TimeStepper used to approximate the time-derivative of the
79 nodal values.
80.
81The meshes used in our example codes (and all the meshes listed below)
82have a common structure that allows the required information
83to be become available to the mesh constructor:
84-# All meshes are templated by the element type, \c ELEMENT.\n\n
85-# The final argument of all mesh constructors is a pointer to a
86 \c TimeStepper. We provide a default for this argument -- a
87 pointer to the \c Steady<0> timestepper, defined (as static
88 member data) in the \c Mesh base class.
89.
90The availability of the template parameter allows the mesh generator
91to build elements of the required type. \c Nodes are generally built by the
92\c FiniteElement::construct_node(...) function whose arguments are
93the \c Node's local node number within the current element,
94and a pointer to the timestepper. These arguments provide all the
95information that is required
96to build \c Nodes with the right number of values (as required by
97the element) and history values (as required by the \c TimeStepper).
98When the function \c FiniteElement::construct_node(...) is called,
99it creates the new \c Node, stores a pointer to the newly created \c Node
100in the \c FiniteElement's own lookup scheme, and returns that pointer.
101This allows the pointer to the newly created \c Node to be stored in
102the \c Mesh's own lookup scheme for its constituent \c Nodes.
103The <A HREF="../../../quick_guide/html/index.html#mesh">
104(Not-So-)Quick-Guide</A> contains a section that explains
105how to write simple meshes.

109\section faq Mesh FAQ
110When using a mesh that was originally developed for a different
111application, it is sometimes necessary establish the node/element/boundary
112numbering scheme employed by the mesh writer. While we generally
113assume that the mesh writer will have carefully documented his/her
114code, here is what to do if he/she hasn't: \n\n
115
116- <B>How do I find out how the elements are numbered?</B> \n\n
117 The function \c Mesh::output(...) outputs the elements in the order in
118 which they are stored internally. If you prefer a different
119 element numbering scheme you can re-number the elements;
120 see e.g. the member function \c element_reorder() in the
121 <A HREF="classoomph_1_1RectangularQuadMesh.html"><B>
122 RectangularQuadMesh</B></A> class.
123 \n\n
124- <B>How do I find out how the nodes are numbered?</B> \n\n
125 The function \c Mesh::node_pt(j) provides pointer-based access to the
126 \c j - th \c Node in the mesh. To plot a node's position, you can
127 determine its coordinates from the \c Node::x(...) function.
128 \n\n
129- <B>How do I find out how the mesh boundaries are numbered?</B> \n\n
130 The function \c Mesh::output_boundaries(...) outputs the nodes located
131 on the mesh boundaries in a
132 <A HREF="http://www.tecplot.com">tecplot</A>-able format. Nodes
133 that are located on separate mesh boundaries are contained in separate
134 <A HREF="http://www.tecplot.com">tecplot</A> zones.
135 \n\n
136.
137
138With this information it should be straightforward to use any
139of the meshes listed below in one of your own problems.
140The example code listed next to each mesh illustrates its use
141in an actual driver code.
142If you develop a new mesh, let us know! If it is written
143according to \c oomph-lib's
144<A HREF="../../../coding_conventions/html/index.html">coding
145standards</A>, we'll be delighted to
146include it into the library.
