3\mainpage Unstructured 2D Delaunay mesh generation with xfig and Triangle
4
5In this document we demonstrate how to use \c oomph-lib's
6conversion routine \c fig2poly (a C++-based stand-alone executable,
7generated from the source code
8<A HREF="../../../../demo_drivers/meshing/mesh_from_xfig_triangle/fig2poly.cc">fig2poly.cc</A>
9to create \c *.\c poly files for <A HREF="http://www.cs.berkeley.edu/~jrs/">
10Jonathan Shewchuk's</A> open-source mesh generator
11<A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle
12</A>, based on the output from the open-source
13drawing program <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig.</A>

18\section xfig_input Mesh generation with xfig, fig2poly and Triangle
19
20Mesh generation with <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A>, \c
21fig2poly and <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">\c Triangle
22</A> is extremely straightforward.
23
24-# Draw the (piecewise linear) domain boundaries, using
25 <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig's</A> polyline
26 drawing tool. Each polyline represents a distinct mesh
27 boundary. The start and end points of the polyline
28 should not coincide -- \c fig2poly
29 will automatically "fill in" the missing line segment.
30 \n\n
31-# If the domain has any holes, place a single circle
32 (drawn with <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig's</A>
33 circle/ellipse drawing tool) into each hole. Use the
34 circle/ellipse drawing tool that requires
35 the specification of the radius. The radius of the circle is
36 irrelevant.
37 \n\n
38-# Save the figure as a \c *. \c fig file.
39 \n\n
40-# Use \c fig2poly to convert the \c *. \c fig file into a \c *. \c
41 fig. \c poly file. For instance,
42 \code
43 ./fig2poly some_figure.fig
44 \endcode
45 will generate a file \c some_figure.\c fig. \c poly. \n\n
46-# Process the \c *. \c fig. \c poly file produced by \c fig2poly with
47 <A HREF="http://www.cs.cmu.edu/~quake/triangle.html">Triangle</A>
48 and use the resulting \c *.poly, \c *.ele and
49 \c *.node files as input to \c oomph-lib's \c TriangleMesh,
50 as described in <A HREF="../../mesh_from_triangle/html/index.html">
51 another example.</A> \n\n
52-# Done!
53.
54
55\subsection xfig_comments Comments:
56
57- \c fig2poly expects the
58 <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A> output file to conform
59 to "Fig format 3.2" and checks for the presence of the string
60 "#FIG 3.2" in the first line of the \c *. \c fig file.
61 below.
62 \n\n
63- The figure must contain only polylines and circles. The presence
64 of any other objects will spawn an error message and cause \c fig2poly
65 to terminate.
66 \n\n
67- The figure must not contain any "compound objects".
68 Compound objects may, of course, be used while drawing but
69 you should break them up before saving the figure.
70 \n\n
71.

158\section comments Comments and Exercises
159\subsection bound Numbering of the mesh boundaries
160-# Each polyline in the <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A>-generated
161 figure represents a distinct mesh boundary. \c fig2poly
162 assigns boundary numbers to these, depending on the
163 order in which the polylines are listed in the \c *. \c fig file.
164 Since this is not always obvious, it is usually necessary to
165 examine the mesh boundaries by calling \c
166 Mesh::output_boundaries(...) before assigning boundary conditions.
167 \n\n
168-# Since boundaries are defined by closed polygons,
169 all nodes that are located on a specific polygon have the
170 same boundary number. In cases where a finer sub-division of the
171 boundary is required (e.g. to identify inflow boundaries)
172 some post-processing of the mesh may be required.
173.
174\subsection ex Exercises
175-# Download and install <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A>
176 (if you work in a linux environment, <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A>
177 is likely to be available already as it is part of many linux
178 distributions) and create your own meshes.
179 \n\n
180-# Think of a way to modify the <A HREF="http://en.wikipedia.org/wiki/Xfig">xfig</A>-based
181 mesh generation procedure so that a closed boundary can contain
182 several sub-boundaries so that in- and outflow boundaries
183 can be identified separately. [This is an "advanced" exercise
184 and one to which no solution exists as yet -- please let us have
185 your code if you find an easy way to do this!].
186.
