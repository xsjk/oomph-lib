3\mainpage Propagation of a droplet in a channel - mesh generation and adaptation for free surface problems
4
5In this tutorial we demonstrate another adaptive solution of free surface
6problems on unstructured meshes, using the example of a droplet
7propagating along a straight channel. The problem is extremely similar
8to <a href="../../adaptive_bubble_in_channel/html/index.html"> the
9propagation of a bubble in a channel tutorial </a>. Thus, we shall
10only discuss the differences from that tutorial. The key physical
11difference is that instead of the uniform pressure state in an
12inviscid bubble, the droplet
13consists of a viscous fluid that can support internal stress variations.
14
15

18\section example The example problem
19We illustrate the solution
20of the unsteady 2D Navier-Stokes equations by considering the
21propagation of a single droplet along a straight channel as shown in
22the sketch below. The non-dimensionalisation is
23the same as in the
24<a href="../../adaptive_bubble_in_channel/html/index.html"> bubble
25tutorial, </a> and we choose the viscosity and density of the
26surrounding liquid to be the reference values.
27
28\image html problem.gif "The problem setup. "
29\image latex problem.eps "The problem setup. " width=0.9\textwidth
30
31The governing equations and boundary conditions are
32the same as those in the
33<a href="../../adaptive_bubble_in_channel/html/index.html"> bubble
34tutorial. </a>
35The only difference is that the Navier--Stokes equations must also be
36solved within the droplet. In fact, this is a two-fluid problem, a
37class of problems that is first introduced in
38<A HREF="../../two_layer_interface/html/index.html"> another tutorial. </A>
39
40<HR>
41
42The constraint that the droplet
43volume remains constant must be enforced explicitly in the static
44case, as in the bubble problem. In the time-simulations, however, a
45constant drop volume is implicitly enforced by the continuity
46equation. For the bubble, the continuity equation is
47not solved within the interior, which is why the volume constraint
48must always be explicitly enforced in that case.

52\section implementation Implementation
53
54We use the same method as in the
55<a href="../../adaptive_bubble_in_channel/html/index.html"> bubble
56problem, </a> an ALE-based finite-element method with
57<a href="../../single_layer_free_surface/html/index.html#kinematic_condition_implementation">a pseudo-elastic
58node-update procedure</a>. In this case,
59there is a pressure jump across the interface between the
60fluids, which means that we use triangular Crouzeix--Raviart
61elements rather than the continuous-pressure Taylor--Hood
62elements. Again, we impose the kinematic and dynamic boundary conditions
63with \c FaceElements. The volume constraint for the static problem
64is also imposed in a similar
65way: we attach \c LineVolumeConstraintBoundingSolidElements to the
66droplet surface and create an additional \c VolumeConstraintElement.
67Here, a pressure degree of
68freedom within the droplet is hijacked, see
69<A HREF="../../static_two_layer/html/index.html"> another tutorial, </A>
70to be used as the unknown associated with the volume constraint. Once
71the static initial problem has been solved, the
72"volume constraint" elements are deleted and the pressure degree of
73freedom is unhijacked.
74

77\section results Results
78We perform the simulation in a two-stage procedure. We start by
79performing a steady solve with the inflow switched off. This
80deforms the droplet into its steady state (approximately) circular
81configuration with the required volume. The actual time-dependent
82simulation is then performed with an impulsive
83start from this configuration.
84
85The figure below shows the location of the droplet and mesh (upper figure) and
86a contour plot of the pressure
87distribution with overlaid velocity vectors of the difference between
88the background Poiseuille flow and the velocity field (lower
89figure). The figure is a snapshot for the parameters \f$ Re = ReSt =
900.0 \f$, \f$ Ca = 10.0 \f$ and a droplet that is ten times as viscous
91as the surrounding liquid.

115\section main The driver code
116
117The first difference from the bubble problem is that the steady solver
118does not converge when \f$ Ca = 10 \f$. Instead, we start with \f$
119Ca = 1 \f$, solve the steady problem, set \f$ Ca=10\f$ and then resolve.
120
121\skipline problem.steady_newton_solve(1)
122\until problem.steady_newton_solve()
123
124After documenting the solution, we remove the (explicit) volume
125constraint and then the remainder of the code is identical to that in
126the <a href="../../adaptive_bubble_in_channel/html/index.html"> bubble
127tutorial. </a>
128
129\skipline Switch off
130\until volume_constraint()

134\section problem The problem class
135
136Other than trivial name changes from "bubble" to "drop", there are
137a few significant changes between this problem and that in
138the
139<a href="../../adaptive_bubble_in_channel/html/index.html"> bubble
140tutorial. </a> There is an additional boolean \c Use_volume_constraint
141and an additional function \c
142remove_volume_constraint(), which are used to manage the switch from
143the explicit enforcement of the volume constraint in the static case
144to the implicit enforcement in the time simulations. The other key
145difference is that we use \c Triangle's
146<a href="../../../meshes/mesh_from_inline_triangle_internal_boundaries/html/index.html#def_extra_regions"> region attributes </a>
147to distinguish the elements inside
148the droplet from those outside. The default behaviour is that all
149elements are in region 0, but we label those element within the drop
150as region 1.

178\section face_elements Generation of face elements
179
180As usual we impose
181the kinematic and dynamic boundary condition at the interface by
182attaching \c FaceElements to the relevant boundaries of the bulk
183elements. However, we must be careful to add only a \b single layer of
184elements. If we use the standard "boundary element" functions then we
185will be creating face elements on both sides of the internal
186boundary. Instead, we use the elements adjacent to the boundary
187within region 0, which ensures that only a single layer of interface
188elements are added.
189
190\dontinclude adaptive_drop_in_channel.cc
191\skipline start_of_create_free_surface_elements
192\until end of create_free_surface_elements
193
194The volume constraint elements are only created if the boolean flag \c
195Use_volume_constraint is true (the default on construction of the problem).
196We hijack the pressure degree of freedom associated with the first
197element in region 1 and then the construction of the elements again
198uses the regions to ensure that a single layer of elements is created.
199
200\skipline start_of_create_volume_constraint_elements
201\until end of create_volume_constraint_elements
202

205\section remove Removal of the volume constraint
206
207The function \c remove_volume_constraint(), resets the boolean flag to
208false, clears the hijacked data, deletes the volume constraint
209elements and mesh and then removes the volume constraint mesh from the
210problem's list of sub meshes, before reassigning the equation numbers.
211
212\dontinclude adaptive_drop_in_channel.cc
213\skipline Change the boundary conditions
214\until }

219\section comments Comments and Exercises
220
221
222 The computation of the initial static solution means that we must
223 still include the calls to the \c create_volume_constraint_elements() and
224 \c delete_volume_constraint_elements() in \c actions_before_adapt()
225 and \c actions_after_adapt(). We have chosen to have the functions
226 return immediately if the volume constraint is not being enforced,
227 rather than using \c if blocks within the \c
228 actions_before/after_adapt() functions.
