3\mainpage Demo problem: 3D Solid Mechanics on unstructured meshes
4
5The purpose of this tutorial is to demonstrate the solution of
63D solid mechanics problems on unstructured meshes.
7The problem studied here also serves as a "warm-up problem" for the
8<a href="../../../interaction/unstructured_three_d_fsi/html/index.html">
9corresponding fluid-structure interaction problem </a> in which the elastic,
10bifurcating vessel whose deformation is studied here conveys (and is
11loaded by) a viscous fluid.
12

17\section problem The problem (and results)
18Here is an animation of the problem: An elastic, bifurcating vessel
19is loaded by an internal pressure and by gravity which acts in the
20negative \f$x\f$-direction. The "ends" of the vessel are held in
21a fixed position. As the magnitude of the loading is
22increased the vessel inflates and sags downwards.
23
24
25\image html deform.gif "An elastic, bifurcating vessel, loaded by an internal pressure and transverse gravity. "
26\image latex deform.eps "An elastic, bifurcating vessel, loaded by an internal pressure and transverse gravity. " width=0.75\textwidth
27
28
29The blue frame
30surrounding the vessel is drawn to clarify its spatial orientation.
31The yellow edges show the boundaries of the
32internal faces via which the pressure loading is applied, and the
33red arrows indicate the direction and magnitude of the pressure loading.

38\section mesh 3D unstructured mesh generation
39We use
40<A HREF="http://www.wias-berlin.de/~si ">Hang Si's</A>
41open-source mesh generator
42<A HREF="http://wias-berlin.de/software/tetgen//"> \c tetgen </A> to generate the
43unstructured tetrahedral mesh "offline". We then
44process the output files produced by
45<A HREF="http://wias-berlin.de/software/tetgen//"> \c tetgen </A>
46to generate an unstructured \c oomph-lib mesh.
47
48<A HREF="http://wias-berlin.de/software/tetgen//"> \c Tetgen </A>
49requires the specification of the domain boundaries
50via so-called facets -- planar surface patches that are bounded
51by closed polygonal line segments. For simplicity, we only
52consider a very simplistic bifurcation, comprising three tube segments
53of approximately rectangular cross-section that meet at a common junction.
54Each of the three tube segments has four internal and four external
55faces. The internal and external faces are connected by three further
56faces at the "in- and outflow" cross-sections (using terminology
57that anticipates the mesh's use in the corresponding
58<a href="../../../interaction/unstructured_three_d_fsi/html/index.html">
59fluid-structure interaction problem</a>), resulting in a total of 27
60facets.
61
62The 27 facets are defined in a <code>*.poly</code> file that
63specifies the position of the vertices, and identifies the
64facets via a "face list" that establishes their bounding vertices.
65Facets that have holes (e.g. the in- and outflow
66facets) require the specification of the hole's position. Finally,
67if the mesh itself has a hole (as in the current example where the
68vessel's lumen forms a hole in the mesh) the position of the hole must be
69identified by specifying the position of a single point inside that
70hole. The well-annotated <code>*.poly</code> file is located at:
71
72<center>
73<a href="../../../../demo_drivers/solid/unstructured_three_d_solid/fsi_bifurcation_solid.poly">demo_drivers/solid/unstructured_three_d_solid/fsi_bifurcation_solid.poly</a>
74</center>
75
76We refer to the <A HREF="http://wias-berlin.de/software/tetgen//"> \c tetgen webpages </A>
77and another <a href="../../../meshes/mesh_from_tetgen/html/index.html">
78\c oomph-lib tutorial</a> for further details on how to
79create <code>*.poly</code> files.
80
81
82Here is a plot of the domain specified by
83<a href="../../../../demo_drivers/solid/unstructured_three_d_solid/fsi_bifurcation_solid.poly">fsi_bifurcation_solid.poly</a>.
84The plot was created using \c tetview which is distributed with
85<A HREF="http://wias-berlin.de/software/tetgen//"> \c tetgen </A>.
86
87\image html tetgen_boundaries.gif "The domain and its bounding facets. "
88\image latex tetgen_boundaries.eps "The domain and its bounding facets. " width=0.75\textwidth
89
90
91Note that we have deliberately assigned a different boundary ID to
92each facet. This will make the assignment of the traction boundary
93condition somewhat tedious as the inner surface of the vessel (where the
94traction is to be applied) is represented by twelve separate
95mesh boundaries. However, the assignment of distinct boundary
96IDs for the different facets is essential for the automatic generation
97of boundary coordinates in the
98<a href="../../../interaction/unstructured_three_d_fsi/html/index.html">
99corresponding fluid-structure interaction problem </a> and is
100therefore <b>strongly recommended</b>.
101
102
103<A HREF="http://wias-berlin.de/software/tetgen//"> \c Tetgen </A> generates
104an unstructured volumetric mesh from the information contained
105in the <code>*.poly</code> file and writes the mesh's nodes,
106elements and faces in the files
107- <a href="../../../../demo_drivers/solid/unstructured_three_d_solid/fsi_bifurcation_solid.1.node">demo_drivers/solid/unstructured_three_d_solid/fsi_bifurcation_solid.1.node </a>
108- <a href="../../../../demo_drivers/solid/unstructured_three_d_solid/fsi_bifurcation_solid.1.ele">demo_drivers/solid/unstructured_three_d_solid/fsi_bifurcation_solid.1.ele </a>
109- <a href="../../../../demo_drivers/solid/unstructured_three_d_solid/fsi_bifurcation_solid.1.face">demo_drivers/solid/unstructured_three_d_solid/fsi_bifurcation_solid.1.face</a>
110.
111These files can be used as input to \c oomph-lib's \c TetgenMesh
112class, using the procedure discussed in
113<a href="../../../meshes/mesh_from_tetgen/html/index.html">
114another tutorial.</a>
115
116 The figure below shows a \c tetview plot of the mesh, created with a volume
117constraint of 0.2 (i.e. the maximum volume of each tetrahedron is
118guaranteed to be less than 0.2 units), using the command
119\code
120tetgen -a0.2 fsi_bifurcation_solid.poly
121\endcode
122
123\image html mesh_boundaries.gif "Plot of the mesh, generated by tetgen. "
124\image latex mesh_boundaries.eps "Plot of the mesh, generated by tetgen. " width=0.75\textwidth
125
126Note how <A HREF="http://wias-berlin.de/software/tetgen//"> \c tetgen </A>
127has subdivided each of the 27 original facets specified in the
128<code>*.poly</code> file into a surface triangulation. The
129nodes and tetrahedral elements that are located on (or adjacent to) the
13027 original facets inherit their boundary IDs. This will be important
131when we assign the boundary conditions.

136\section mesh_code Creating the mesh
137
138We create the solid mesh by multiple inheritance from \c oomph-lib's
139\c TetgenMesh and the \c SolidMesh base class:
140
141\dontinclude unstructured_three_d_solid.cc
142\skipline start_mesh
143\until {
144
145The constructor calls the constructor of the underlying
146\c TetgenMesh (using the <code>*.node </code>
147<code>*.ele</code> and <code>*.face</code> files created
148by <A HREF="http://wias-berlin.de/software/tetgen//"> \c tetgen. </A>
149As usual we set the nodes' Lagrangian coordinates to their current
150Eulerian positions, making the current configuration
151stress-free.
152
153\until set_lagrangian_nodal_coordinates()
154
155Finally, we identify the elements that are located next to the
156various mesh boundaries to facilitate the application of the
157traction boundary conditions.
158
159\until };

164\section namespace Problem parameters
165As usual, we define the various problem parameters in a
166global namespace. We use \c oomph-lib's generalised Hookean
167constitutive law as the constitutive equation, using a Poisson's
168ratio of 0.3.
169(<a href="../../solid_theory/html/index.html#non-dim_solid"> Recall</a>
170that omitting the specification of Young's modulus, \f$ E \f$,
171implies that the stresses are non-dimensionalised on \f$ E \f$ .)
172
173
174\skipline start_namespace
175\until Constitutive_law_pt
176
177Next we define the gravitational body force which acts in
178the negative \f$x\f$-direction,
179
180\until }
181
182and the pressure load, \f$ {\bf t} = -P {\bf n} \f$, that acts on
183internal walls of the bifurcation (note that the outer unit
184normal \f$ {\bf n} \f$ on the wall is passed to the
185function).
186
187
188\until end namespace

193\section main The driver code
194
195The driver code is straightforward. We store the command line
196arguments, specify an output directory and create the problem object,
197using ten-noded tetrahedral solid mechanics elements to
198discretise the principle of virtual displacements.
199
200\dontinclude unstructured_three_d_solid.cc
201\skipline start_main
202\until TPVD
203
204We output the initial configuration and then perform a parameter
205study in which we increment the gravitational body force and the pressure
206loads simultaneously, causing the vessel to expand and
207sag, as shown in the animation at the beginning of this tutorial.
208(As usual we perform a smaller number of steps if the
209code is run in self-test mode; this is indicated by a non-zero number
210of command line arguments.)
211
212\until end main

218\section class The Problem class
219The \c Problem class has the usual member functions, and provides
220storage for the various sub-meshes -- the bulk mesh of 3D solid elements
221and the meshes of 2D solid traction elements that apply the
222pressure load to the internal boundaries of the vessel.
223
224\dontinclude unstructured_three_d_solid.cc
225\skipline start_problem
226\until Solid_traction_mesh
227
228
229The two vectors \c Pinned_solid_boundary_id and
230\c Solid_traction_boundary_id are used to store the IDs of mesh boundaries
231that make up the in- and outflow cross-sections (where the
232bifurcation is pinned), and the internal boundaries (where
233the pressure load has to be applied). Recall that \c tetgen requires the domain
234boundaries to be specified as a collection of planar facets. Boundary
235of interest in the computation, such as the "internal boundary of the
236bifurcation", therefore tend to comprise multiple distinct mesh
237boundaries.
238
239\until };

270

276
277\until bc_file.close();

328\subsection bound Identification/assignment of mesh boundaries
329This tutorial shows that the use of unstructured 3D meshes
330for solid mechanics problems is extremely straightforward.
331The only aspect that requires some care (and not just for solid
332mechanics applications) is the correct identification/assignment of
333domain boundaries. The fact that we documented the position of the
334pinned nodes in the driver code suggests (correctly!) that we managed
335to get both assignments (slightly) wrong when we first generated
336the mesh and wrote the corresponding driver code. As usual,
337it pays off to <B>be as a paranoid as possible</B>! Ignore this
338advice at your own risk...
