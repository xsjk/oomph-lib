3\mainpage Refineable Boussinesq Convection using a multi-domain approach: Combining (Refineable) Navier--Stokes and Advection-Diffusion domains
4
5In this tutorial we present an alternative approach to the solution
6of multi-physics problems, using what we call a "multi-domain"
7approach. We illustrate the methodology
8by re-considering the <a href="../../b_convection/html/index.html">
9previously-described, </a>
10two-dimensional Boussinesq convection problem in
11which an incompressible, Newtonian fluid is heated from below.
12
13The tutorial is quite long but comprises four relatively self-contained parts.
14Feel free to skip the theory if you want to jump straight in...
15\n\n
16- <a href="#setting_up"><b>Part I</b></a> provides an overview of \c oomph-lib's
17 overall framework for setting up multi-domain interactions.
18 \n\n
19- <a href="#implement_boussinesq"><b>Part II</b></a> illustrates the general
20 procedure for the specific example of a Boussinesq convection problem. We
21 demonstrate how to implement a two-way, multi-domain interaction between
22 the Navier-Stokes equations in which the body force is affected by thermal
23 buoyancy effects, and the advection-diffusion equations for the
24 temperature, in which the "wind" is given by fluid velocity.
25 \n\n
26- <a href="#multi_domain_boussinesq"><b>Part III</b></a> discusses the
27 driver code to solve the Boussinesq problem, using the elements developed
28 in Part II.
29 \n\n
30- <a href="#optimise"><b>Part IV</b></a> discusses how to optimise
31 multi-domain interactions.
32.

36\section setting_up Part I: The overall framework for handling multi-domain interactions
37
38
39
40When two or more physical processes interact within the same spatial
41domain there are three different approaches to setting up the
42interaction within \c oomph-lib. One approach is to write a completely
43new element that discretises all the PDEs involved in the problem.
44The second approach, described in the tutorials discussing the
45<a href="../../b_convection/html/index.html"> the non-refineable </a> and
46<a href="../../refine_b_convect/html/index.html">refineable </a>
47solution of the Boussinesq convection problem, is to create a
48combined multi-physics element using inheritance from two, or more, existing
49elements. In both these cases, the physical processes and their
50associated fields interact within a single element and we refer
51to this methodology as a \b multi-field approach.
52
53In this tutorial we describe a third approach to multi-physics
54problems, in which we solve the problem using two different types of elements
55on two different meshes (domains), each occupying the same
56physical space. Rather than interacting locally
57within elements, the different physical processes interact
58directly between the two domains, so that, in
59the Boussinesq convection problem considered here, each
60advection-diffusion element obtains the "wind" from the (separate)
61Navier-Stokes element that occupies the same position, while
62the Navier-Stokes elements obtain the temperature
63required for the computation of the buoyancy force from the
64corresponding advection-diffusion element.
65
66
67One benefit of this \b multi-domain approach, compared to a \b
68multi-field approach, is that different error estimators can be
69used on each domain and there is no longer a need to construct
70a combined multi-physics error estimator; see the
71<a href="../../refine_b_convect/html/index.html">tutorial for the
72adaptive solution of the Boussinesq convection problem</a> for more
73details on this issue. Moreover, the meshes do not
74have to have the same refinement pattern, which can be
75advantageous if the different physical processes act over different
76spatial scales. The figure below shows that in the Boussinesq
77convection problem the Navier--Stokes mesh (panel (a)) requires much more
78refinement than the advection-diffusion mesh (panel (b)).
79
80\image html two_domains.gif "Steady Convection Rolls: (a) contours of x-velocity and corresponding element boundaries in Navier--Stokes mesh, (b) contours of temperature and corresponding element boundaries in advection-diffusion mesh. "
81\image latex two_domains.eps "Steady Convection Rolls: (a) contours of x-velocity and corresponding element boundaries in Navier--Stokes mesh, (b) contours of temperature and corresponding element boundaries in advection-diffusion mesh. " width=0.75\textwidth

86\subsection external_elements The ElementWithExternalElement class
87
88 Interaction between different elements in different domains is a
89fundamental feature of many multi-physics problems and the generic
90functionality to deal with such interactions is provided by the \c
91ElementWithExternalElement class. Any element that requires
92information from an "external element" should therefore inherit from the
93base class \c ElementWithExternalElement.
94
95"External elements" typically provide source-like terms for the
96\c ElementWithExternalElement. For instance, in the Boussinesq
97convection problem studied here,
98the advection-diffusion element is the "external element" for
99the Navier-Stokes element because it affects the body force
100in the Navier-Stokes equations; in a fluid-structure interaction
101problem, the Navier-Stokes elements that are adjacent to the
102fluid-loaded elastic wall act as the "external elements" for the
103\c FSIWallElement because they determine the traction that the fluid
104exerts onto the wall; etc.
105
106 Within a finite-element context, the effect of the "external element"
107onto the \c ElementWithExternalElement typically has to be evaluated
108at the integration points of the \c ElementWithExternalElement.
109Therefore, the \c ElementWithExternalElement base class provides
110storage for a pointer to an "external element" (and the local coordinate
111within it) for each integration point. Access to this information
112is provided via the member functions
113
114
115\code
116FiniteElement*& ElementWithExternalElement::external_element_pt(
117 const unsigned& interaction,
118 const unsigned& ipt);
119\endcode
120
121and
122
123\code
124Vector<double>& ElementWithExternalElement::external_element_local_coord(
125 const unsigned& interaction,
126 const unsigned& ipt);
127\endcode
128
129For a given integration point, \c ipt, within the \c
130ElementWithExternalElement, these functions
131identify the corresponding point in the "external element" by
132returning a pointer to the "external element" and the local coordinate
133within it, respectively.
134The unsigned parameter \c interaction denotes the interaction
135index, required to allow for cases where an element interacts with more
136than one "external element". This occurs, for instance, in double-diffusive
137convection problems (discussed in <a href="../../../../demo_drivers/multi_physics/double_diffusive_convection/multimesh_dd_convection.cc">
138a separate demo driver code </a>), in which the body force
139in the Navier-Stokes equations is affected by two physical properties,
140<em> e.g.</em> temperature and salinity, both of which are transported by
141separate advection-diffusion processes;
142or in FSI problems where a beam or shell element is
143affected by a fluid load on both its "front" and "back" (see, e.g., the
144<a href="../../../interaction/fsi_channel_with_leaflet/html/index.html">FSI
145channel with leaflet problem</a>).
146
147\c Data that affects the fields in the "external elements"
148must be specified so that the off-diagonal blocks in the
149\c ElementWithExternalElement's Jacobian matrix (representing the
150derivatives of the \c ElementWithExternalElement's residual vector
151with respect to the \c Data in the "external elements")
152can be calculated. We divide such data into two categories:
153- <em> External field data: </em> all \c Data that affects the
154fields through which the "external element" interacts with the
155\c ElementWithExternalElement.
156.
157- <em> External geometric data: </em> all geometric \c Data that affects
158the shape and position of the "external element" and, therefore, spatial
159derivatives of its fields.
160.
161The rationale for this division is that in many cases the field
162data's contribution to the interaction blocks in the Jacobian matrix
163can be found analytically, whereas the geometric
164data is more easily handled by finite differencing.
165
166Both types of external data are determined (automatically) by two member
167functions of the \c ElementWithExternalElement. The first one,
168
169\code
170virtual void ElementWithExternalElement::identify_all_field_data_for_external_interaction(
171 Vector<std::set<FiniteElement*> > const &external_elements_pt,
172 std::set<std::pair<Data*,unsigned> > &paired_interaction_data);
173\endcode
174
175determines the external field data that affect the (possibly multiple)
176interactions in the \c ElementWithExternalElement. Given the vector
177\c external_elements_pt (within which <code> external_elements_pt[i] </code>
178contains the set of pointers to the "external elements"
179involved in the \c ElementWithExternalElement's \c i -th interaction), the
180function returns a set of pairs, each of which comprises a
181pointer to a \c Data object and an unsigned that identifies a value
182within it. The default implementation of this function includes
183\b all field data of \b all "external elements", using calls to
184
185\code
186void FiniteElement::identify_field_data_for_interactions(...)
187\endcode
188
189for each "external element". Similarly, the function
190
191\code
192virtual void ElementWithExternalElement::identify_all_geometric_data_for_external_interaction(
193 Vector<std::set<FiniteElement*> > const &external_elements_pt,
194 std::set<Data*> &external_geometric_data_pt);
195\endcode
196
197returns the set of \b all \c Data that affect the shape or position of any
198of the "external elements" specified by the vector \c
199external_elements_pt. Again, this function has a default
200implementation that determines \b all geometric Data associated
201with "external elements" by calling
202
203\code
204void FiniteElement::identify_geometric_data(..)
205\endcode
206
207for each "external element".
208
209We note that the default implementation of these functions can
210(and, where possible, should) be overloaded to exclude data values
211that do not actually contribute to the interaction. For instance,
212in the Boussinesq convection problem considered here, the
213advection-diffusion elements are only affected by the
214velocity degrees of freedom in the Navier-Stokes elements but
215not by the pressures. Our re-implementation of \c
216ElementWithExternalElement::identify_all_field_data_for_external_interaction(...)
217in the multi-domain advection-diffusion element for the Boussinesq problem
218therefore excludes the pressure degrees of freedom, see
219<a href="#optimise"><b>Part IV</b></a>.

223\subsection off_diag_jac Computation of the Jacobian matrix
224
225Most single-physics elements already provide functions
226to compute their own Jacobians analytically. These functions compute the
227derivatives of the element's residual vector with respect to the
228element's "own" degrees of freedom (e.g. the fluid velocity and pressure in a
229Navier-Stokes element). The interaction with the "external element"
230introduces additional dependencies because the element's residual vector
231now also depends on the unknowns associated with the "external
232elements". The derivatives of the element's residual
233vector with respect to these unknowns must therefore be included into the
234computation of the element's Jacobian matrix.
235
236
237To maximise code re-use, we follow the approach discussed in
238the <a href="../../b_convection/html/index.html#comments">
239comments </a> section of the single-domain version of the problem
240and re-use the underlying element's \c fill_in_contribution_to_jacobian(...)
241function, while employing finite-differencing to evaluate the derivatives
242of the element's residual vector with respect to the unknowns
243associated with the "external elements". This is done most easily by
244using the function
245
246\code
247ElementWithExternalElement::fill_in_jacobian_from_external_interaction_by_fd(..)
248\endcode
249
250which computes the derivatives of the element's
251residual vector with respect to external field and external geometric
252data by calls to the two functions
253
254\code
255ElementWithExternalElement::fill_in_jacobian_from_external_interaction_field_by_fd(..)
256\endcode
257
258and
259
260\code
261ElementWithExternalElement::fill_in_jacobian_from_external_interaction_geometric_by_fd(..)
262\endcode
263
264If, in a particular problem, it is known that the interaction is not
265affected by the position or shape of the "external elements" (<em>
266i.e. </em> if
267the interaction terms do not involve spatial derivatives of the
268"external element's" field variables) the unnecessary computation of the
269derivatives with respect to the "external element's" geometric data
270may be suppressed using a call to
271
272\code
273ElementWithExternalElement::ignore_external_geometric_data();
274\endcode
275
276Depending on the complexity of the interaction terms, it may be
277worthwhile to provide a function that computes the off-diagonal
278blocks analytically. This is discussed in more detail in
279<a href="#optimise"><b>Part IV</b></a>.

284\subsection setting_up_interaction Setting up the interaction
285
286The namespace \c Multi_domain_functions provides numerous helper
287functions that facilitate the setup of multi-domain interactions.
288Specifically, the function
289\c Multi_domain_functions::setup_multi_domain_interactions(...)
290can be used to identify the "external elements" in a two-way
291interaction between two meshes. The function has the following
292interface:
293
294\code
295 template<class ELEMENT_0,class ELEMENT_1>
296 void setup_multi_domain_interactions(Problem* problem_pt,
297 Mesh* const &first_mesh_pt,
298 Mesh* const &second_mesh_pt,
299 const unsigned& first_interaction=0,
300 const unsigned& second_interaction=0);
301\endcode
302
303Here \c first_mesh_pt and \c second_mesh_pt point to the two
304interacting meshes whose elements (of type \c ELEMENT_0 and
305\c ELEMENT_1, respectively), must be derived
306from the \c ElementWithExternalElement class.
307The optional interaction parameters may be used to specify
308which interaction is set up in each mesh. If the parameter is not specified
309it defaults to zero, appropriate if there is
310only a single interaction.
311
312The function must be called prior to the assignment of the equation
313numbers, and then again whenever either of the two meshes has changed,
314<em> e.g.</em> after a mesh adaptation.

584\section optimise Part IV: Optimising multi-domain interactions
585We showed in <a href="#implement_boussinesq"><b>Part II</b></a>
586that only a small number of functions <b> must </b> be implemented to upgrade
587an existing single-physics element to an \c ElementWithExternalElement
588that can interact with another element in a different domain. This is
589because much of the required functionality is already implemented
590in the \c ElementWithExternalElement base class, which makes
591the implementation of multi-physics interactions very easy. The
592use of general-purpose functions may incur unnecessary computational
593cost, however. Hence, a better efficiency can be achieved by
594overloading certain functions when implementing a specific
595\c ElementWithExternalElement.
596
597<HR>
598
599\subsection restricting_field_data Ignoring field data that does not participate in the interaction
600
601The function
602\c ElementWithExternalElement::identify_all_field_data_for_external_interaction(...)
603assumes that \b all data values in the external elements affect
604the interaction with the \c ElementWithExternalElement. In many
605cases this assumption is overly pessimistic (and costly). For
606instance in the Boussinesq convection problem only the velocity
607degrees of freedom in the (external) Navier-Stokes element affect
608the the advection-diffusion equations via the "wind". It is therefore
609sensible to exclude the pressure degrees of freedom from the
610interaction by re-implementing
611\c identify_all_field_data_for_external_interaction(...) as follows
612
613\dontinclude multi_domain_boussinesq_elements.h
614\skipline optimised_identification_of_field_data
615\until // done
616
617Similar approaches can be used to ignore selected (weak) interactions
618when computing the element's Jacobian matrix. For instance,
619the \c FSIWallElement is an \c ElementWithExternalElement
620for which the adjacent fluid elements that apply the fluid
621traction to the FSI boundary act as "external elements".
622In high-Reynolds-number flows, the fluid traction is dominated
623by the pressure while shear stresses tend to be small. The \c FSIWallElement
624therefore uses the same mechanism as illustrated above to (optionally)
625neglect the derivatives of its residuals with respect to the fluid
626velocity degrees of freedom. We stress that this does not exclude
627the shear stress from the computation -- it simply replaces the
628exact Jacobian by an approximate version in which the effect
629of the velocity degrees of freedom on the residuals of the \c
630FSIWallElement are neglected. This may lead to a slight degradation
631in the convergence rate of the Newton iteration but this may be
632more than compensated for by the reduction in the CPU times required to
633compute the Jacobian matrix.
634
635<HR>
636
637\subsection restricting_geom_data Ignoring geometric data
638
639If the interaction with the "external element" does not involve
640spatial derivatives of the fields represented by the "external element"
641or if the mesh containing the "external elements" is fixed, the
642external geometric data associated with the "external elements" can
643be ignored when computing the \c ElementWithExternalElement's
644Jacobian. This may be achieved by calling
645
646\code
647ElementWithExternalElement::ignore_external_geometric_data();
648\endcode

652\subsection analytical_jacobian Computing the off-diagonal blocks in the Jacobian analytically
653
654The setup of the Jacobian matrix can be made much more efficient
655by computing the derivatives of the \c ElementWithExternalElements'
656residual vector with respect to the field data in the associated
657"external elements" analytically. The multi-domain driver codes in
658
659<CENTER>
660<A HREF="../../../../demo_drivers/multi_physics/boussinesq_convection">
661demo_drivers/multi_physics/boussinesq_convection
662</A>
663</CENTER>
664
665demonstrate a possible implementation. The key challenge for the
666implementation is that the \c ElementWithExternalElement must label the entries
667in its elemental Jacobian matrix by its local equation
668numbers, whereas the "external element" can only compute the
669derivative of its fields with respect to its own (differently numbered)
670local degrees of freedom (d.o.f.s). To establish which local d.o.f. in the
671"external element" corresponds to a given local d.o.f.
672in the \c ElementWithExternalElement we exchange the (unique) global
673equation numbers associated with each d.o.f. Using this trick, the
674computation of the derivatives becomes relatively straightforward.
675Mathematically, it involves repeated applications of the chain rule.
676In the cases we considered, the fully-analytic computation of the
677elemental Jacobian matrix was about 3 to 4 times faster than
678finite-difference-based computation implemented in the
679\c ElementWithExternalElement base class. Whether the speedup is
680worth the additional (human) time required to implement the
681analytic computation of the off-diagonal entries depends on
682the application (and you!). If nothing else, the availability of a
683finite-difference based routine helps in the validation of any
684newly-developed analytic re-implementation.

690\section comm_and_ex Comments and Exercises
691\subsection comments Comments
692- The use of two separate meshes with different error estimators means
693that, in principle, a more accurate solution can be obtained with
694fewer degrees of freedom than using a combined error estimator on a
695single mesh. The combined error estimator implemented in the
696<a href="../../refine_b_convect/html/index.html">single-domain
697version of the problem </a> will cause refinement if either the fluid
698error or the advection-diffusion error is above the
699specified tolerance. Thus, a fully-converged solution in the
700single-domain problem is one in which the estimated error in
701all field variables is below the tolerance, but this may be at the
702cost of some over-refinement in one or more of the field variables. In
703the present multi-domain approach, if the solution is converged on
704both meshes then the estimated error in all field variables is again
705below the tolerance. Hence, it is fair, in some sense, to
706compare the results between fully-converged
707solutions for the single- and multi-domain problems. As suggested by
708the figure above, for the Boussinesq convection problem
709considered here we do obtain fully-converged solutions
710with fewer degrees of freedom using the multi-domain approach because
711the temperature field is over-refined in the single-domain
712case. Of course, the multi-domain solution includes the extra overhead
713of setting up the interaction, but, in general, this cost is
714negligible compared to the solution of the linear systems.
715\n\n
716- The general procedures described in this tutorial can be used to
717set up any interaction between different types of elements using
718multiple meshes.
719\n\n
720- Before any refinements take place the combined Jacobian
721is exactly the same as that in the single-domain problem and
722so the residuals at each Newton step will be exactly the same. You can
723verify this by comparing the appropriate output files in the
724\c Validation directory.
