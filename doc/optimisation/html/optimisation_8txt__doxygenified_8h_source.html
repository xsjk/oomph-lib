3\mainpage Optimisation
4
5One of the main challenges in the development of a
6general-purpose scientific computing library is
7the potential conflict between the desire for
8robustness and generality on the one hand, and code optimisation
9(in terms of speed and memory requirements) on the other.
10
11One of \c oomph-lib's main design principles is that
12
13<center>
14<EM><B>Robustness/correctness is more important than "raw speed",</B></EM>
15</center>
16
17implying that \e by \e default, all methods should reliably produce
18the \e correct results. Possible optimisations that may lead to
19incorrect results even for a small subset of cases, must be
20activated explicitly by the "knowledgeable" user, who then
21takes the responsibility for any resulting errors.
22
23Here are some examples for possible optimisations that you may wish
24to activate for your problem.
25- \ref linear_vs_nonlinear
26- \ref store_shape
27- \ref full_integration
28- \ref ale
29- \ref C_style_output
30- \ref assembly
31.
32All tend to lead to faster run times
33(by how much depends on your machine and your compiler --
34experiment!), but there are caveats that you must be aware of:
35- Some techniques may incur significant memory overheads (e.g. storing
36 the shape functions).
37- Some techniques may produce the wrong results for your problem
38 (for instance, declaring a problem to be linear when it isn't
39 will lead to a very significant speedup. However, the results will
40 be wrong!)
41- Some techniques will make the code less robust (e.g. using
42 C-style output).
43.
44You choose!

53\section linear_vs_nonlinear Linear vs. nonlinear problems
54By default, \c oomph-lib treats all problems as nonlinear
55and uses Newton's method to solve the system of nonlinear algebraic
56equations that result from the problem's discretisation. With this
57approach, linear problems are special cases for which Newton's method
58converges in one step. However, if a problem is known to be linear,
59several (costly) steps in Newton's method are superfluous and may
60be omitted, leading to a significant speedup. For instance, in a linear
61problem it is not necessary to re-compute the residuals of the nonlinear
62algebraic equations after the (single) Newton step, as we already know that
63they will have been reduced to zero (modulo roundoff errors).
64We refer to the document
65<A HREF="../../order_of_action_functions/html/index.html">"Action"
66functions in oomph-lib's black-box Newton solver</A> for a more detailed
67discussion.
68
69The user may declare a problem to be linear by setting the flag
70\code
71bool Problem::Problem_is_nonlinear
72\endcode
73which is initialised to \c true in the constructor of the \c Problem
74base class, to \c false.
75
76Experiment with the demo code
77<A HREF="../../../demo_drivers/optimisation/linear_vs_nonlinear/two_d_poisson.cc">two_d_poisson.cc</A>
78to examine the speedup achievable by declaring a (linear!)
79Problem to be linear.

84\section store_shape Storing the shape functions
85If a problem is to be solved repeatedly (e.g. in a time-dependent
86simulation, or during a parameter study) it may be beneficial
87to avoid the re-computation of the shape function and their
88derivatives with respect to the local and global coordinates at the
89elements' integration points by storing their values.
90
91 However, there are two problems with this approach:
92-# It introduces significant memory overheads. For instance, to
93 store just the shape functions in a 3D problem discretised by
94 \f$ N_{elem}\f$ 27-node brick elements
95 with a 27-point Gauss rule, storage for \f$ 27 \times 27 \times
96 N_{elem} \f$ doubles is required. Storage of the (much more
97 costly to compute) shape function derivatives requires
98 six times that storage.
99-# The most costly-to-compute quantities are the derivatives
100 of the shape functions with respect to the global
101 coordinates, so it is most tempting to store these.
102 However, in free-boundary problems in which the nodal
103 positions are determined as part of the solution, the
104 derivatives of the shape functions with respect
105 to the global coordinates depend on the solution and \b must
106 be recomputed.
107.
108To ensure correctness/robustness and to minimise the memory overheads,
109\c oomph-lib's existing finite elements represent the shape
110functions and their derivatives as \c StorableShapeFunction
111objects. These allow their values to be stored at the integration
112points, but the capability is not activated, unless an element
113(of type \c ELEMENT, say) is used in its "wrapped"
114form, as \c StorableShapeElement<ELEMENT>.
115
116The detailed documentation for this is yet to be written
117but you may consult the well-documented demo code
118<A HREF="../../../demo_drivers/optimisation/stored_shape_fcts/two_d_poisson_stored_shape_fcts.cc">two_d_poisson_stored_shape_fcts.cc</A>,
119and the discussion in the
120<A HREF="../../quick_guide/html/index.html#pre_compute_psi">
121(Not-So-)Quick Guide.</A>

126\section full_integration Full vs. reduced integration
127To maximise the potential for code reuse, all existing finite elements
128in \c oomph-lib are composed (via multiple inheritance)
129from (i) "geometric" finite elements (e.g. line, quad, brick, triangle and
130tet elements), and (ii) separate equations classes. A default (spatial)
131integration scheme is defined for each geometric element. This default
132integration scheme is chosen to provide "full integration" for this
133element. This implies that for an undeformed element the
134products of any two shape functions are integrated exactly.
135For time-dependent problems (e.g. in
136problems involving the unsteady heat equation) where products
137of shape functions arise in the "mass matrix", "full integration"
138is required to ensure that the numerical solution converges to the exact
139solution under mesh refinement. In some applications
140(e.g. in problems involving the Poisson equation) a lower-order
141(and therefore cheaper) "reduced integration" scheme may be sufficient
142to maintain the asymptotic convergence rate.
143
144In this case, the user can over-write the default spatial integration
145scheme, using the function \c
146FiniteElement::set_integration_scheme(...),
147as illustrated here.
148\code
149
150 // Create an instance of a lower-order integration scheme
151 Gauss<2,2>* int_pt=new Gauss<2,2>;
152
153 //Find number of elements in mesh
154 unsigned n_element = problem.mesh_pt()->nelement();
155
156 // Loop over the elements
157 for(unsigned i=0;i<n_element;i++)
158 {
159 //Set the a different integration scheme
160 problem.mesh_pt()->finite_element_pt(i)->set_integration_scheme(int_pt);
161 }
162
163\endcode
164
165Driver codes that may be used to experiment with reduced integration
166and to assess the asymptotic convergence behaviour can be found
167in the directory
168<A HREF="../../../self_test/poisson/convergence_tests">self_test/poisson/convergence_tests/</A>

172\section ale Disabling the ALE formulation of unsteady equations
173Most existing finite elements for unsteady problems are
174formulated in the "Arbitrary Lagrangian Eulerian (ALE)" form
175by implementing the Eulerian time-derivative \f$ \partial u /\partial
176t\f$ (to be evaluated at a fixed Eulerian position)
177as
178\f[
179\left. \frac{\partial u}{\partial t} \right|_{\mbox{fixed Eulerian position}}
180=
181\left. \frac{\partial u}{\partial t} \right|_{\mbox{fixed local
182coordinate}}
183- \sum_{i=1}^{DIM} v_i^{\mbox{[Mesh]}} \ \frac{\partial u}{\partial x_i}
184\f]
185where \f$ v_i^{\mbox{[Mesh]}} \ \ (i=1,...,DIM)\f$ is the
186mesh velocity. This formulation ensures that the time-derivatives
187are computed correctly if the element is used in a free-boundary value
188problem. However, the computation of the mesh velocities introduces
189an additional cost which is avoidable if the mesh is, in fact, stationary.
190The user may disable the computation of the
191ALE terms by calling the function
192\code
193FiniteElement::disable_ALE()
194\endcode
195(This function is implemented as a virtual function in the
196\c FiniteElement base class. If called, it simply issues a warning
197message to announce that it has not been overloaded in a derived
198class -- usually an indication that the element in question
199does not involve any ALE terms). The ALE capability may be
200re-enabled by calling the corresponding function
201\code
202FiniteElement::enable_ALE()
203\endcode
204
205Experiment with the driver codes in the directory
206<A HREF="../../../demo_drivers/optimisation/disable_ALE">demo_drivers/optimisation/disable_ALE</A>
207to examine the speedup achievable by ignoring the ALE terms in
208the
209<A HREF="../../../demo_drivers/optimisation/disable_ALE/unsteady_heat">unsteady
210heat</A> and
211<A HREF="../../../demo_drivers/optimisation/disable_ALE/navier_stokes">Navier-Stokes</A> equations.

216\section C_style_output C vs. C++-style output
217In most <A HREF="../../example_code_list/html/index.html">demo driver
218codes</A> post-processing is performed by passing a C++-stream
219to a high-level output function, as in this code segment
220\code
221 #include<ofstream>
222
223 [...]
224
225 ofstream output_stream("solution.dat");
226 mesh_pt()->output(output_stream);
227 output_stream.close();
228
229 [...]
230
231\endcode
232
233While C++-streams are very convenient, they can be expensive -- so
234much so that in extreme cases a computation can become dominated by
235the cost of the post-processing operations. There is no obvious reason
236why I/O with C++ streams should be so much slower than I/O in fortran or C,
237say. In fact, it is not supposed to be! (Google will lead you to numerous
238heated discussions on this topic.) However, we (and, it seems, many
239others) have found C++-output to be consistently slower than C-style
240output. Since the latter is available from C++ we have started to
241provide alternative output routines that employ C-style I/O,
242using the FILE type. If your computation involves lots of
243I/O, it may be helpful to replace the above statements by
244
245
246\code
247 #include<stdio.h>
248
249 [...]
250
251 FILE* file_pt = fopen("solution.dat","w");
252 mesh_pt()->output(file_pt);
253 fclose(file_pt);
254
255 [...]
256
257\endcode
258
259Depending on the application, we typically find that C-style output
260is about 2 to 6 times faster than its C++ equivalent. Run the demo
261code
262<A HREF="../../../demo_drivers/optimisation/C_style_output/c_style_output.cc">c_style_output.cc</A>
263to explore the relative performance of the two methods on your machine.
264However, C-style output is not as "bullet-proof" as its
265C++-counterpart. For instance, if an output directory does not exist,
266trying to write to it with C-style output tends to cause a segmentation
267fault, whereas C++-output handles this more gracefully (no output
268is produced but the code execution continues).

274\section assembly Different sparse assembly techniques and the STL memory pool
275
276\subsection assembly_methods The different sparse assembly methods and how to compare their relative performance
277
278\c oomph-lib's black-box nonlinear solver
279\c Problem::newton_solve() employs Newton's method to solve the
280system of nonlinear algebraic equations arising from the
281discretisation of the problem. By default, the linear systems
282that determine the corrections to the unknowns during the Newton
283iterations are solved with \c SuperLUSolver, \c oomph-lib's default
284\c LinearSolver. Within this framework the two most computationally
285expensive steps (both in terms of memory usage and CPU time) are the
286assembly of the linear system (comprising the Jacobian matrix and
287the residual vector) and its solution. [Not all
288of \c oomph-lib's \c LinearSolvers require the assembly of the
289Jacobian matrix (for instance the frontal solver \c HSL_MA42
290avoids the assembly of the linear system and computes its LU
291decomposition "on the fly") but most of them do.]
292
293 The assembly of the Jacobian matrix (in compressed-row or
294compressed-column storage) is typically performed by the function
295
296\code
297Problem::sparse_assemble_row_or_column_compressed(...)
298\endcode
299
300This function performs the assembly, using one of five
301different assembly methods, selected by the protected member data
302
303\code
304unsigned Problem::Sparse_assembly_method
305\endcode
306
307in the \c Problem class. This flag can take any of the
308values defined in the enumeration \c Problem::Assembly_method.
309
310Each of the five different methods has its own advantages and
311disadvantages in terms of speed and/or memory requirements.
312By default, we use the "assembly using vectors of pairs"
313as it tends to be optimal in both respects --
314at least when the library is compiled with full optimisation.
315If the library is compiled without optimisation and/or in \c PARANOID
316mode, the map-based assembly tends to be faster, if significantly
317more expensive in terms of memory requirements.
318
319 You should explore the performance of the various
320methods on your machine, using the the driver code
321<A HREF="../../../demo_drivers/optimisation/sparse_assemble/sparse_assemble_test.cc">sparse_assemble_test.cc</A>
322in the directory
323
324<CENTER>
325<A HREF="../../../demo_drivers/optimisation/sparse_assemble/">
326demo_drivers/optimisation/sparse_assemble</A>
327</CENTER>
328
329The shell script <A HREF="../../../demo_drivers/optimisation/sparse_assemble/compare.bash">
330compare.bash</A> in the same directory
331may be used to systematically explore the performance of
332the different methods for various problem sizes.
333
334While the test code automatically documents the CPU times required
335for the matrix assembly, the automatic assessment of its
336memory usage is a somewhat nontrivial task. We provide a
337quick-and-dirty solution to this problem: If the protected
338member data
339
340\code
341bool Problem::Pause_at_end_of_sparse_assembly
342\endcode
343
344is set to true, the assembly process is halted when it has reached
345its most memory-intensive phase. The user can then use
346\c top (or some other external tool) to assess the
347memory usage of the code. (Typing "M" in \c top sorts the entries
348by memory usage rather than by cpu usage).

353\subsection STL_pool The STL memory pool
354
355When analysing the code's memory usage, you may notice that, following
356the assembly of the Jacobian matrix, not all memory is returned to
357the system, particularly, if maps or lists are used for the assembly.
358This does \b not indicate the presence of a memory leak but is a
359result of the internal memory management performed by the STL
360allocator which retains some (and in some cases a lot) of the
361initially allocated memory in its own "memory pool" in order to speed
362up the subsequent memory allocation for other STL objects.
363<A HREF="http://www.google.com">Google</A> for "STL memory pool"
364to find out more about this issue. The executive
365summary is: "Memory pools are good for you", even though they have
366the slightly annoying side-effect of making it virtually impossible
367to monitor the \e actual memory usage of the code.
