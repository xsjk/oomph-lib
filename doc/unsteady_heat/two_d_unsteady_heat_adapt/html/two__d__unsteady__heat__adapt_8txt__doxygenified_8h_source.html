3\mainpage Example problem: Spatially adaptive solution of the 2D unsteady heat equation with flux boundary conditions.
4
5
6This is a slightly more advanced example in which we demonstrate
7the use of spatial adaptivity in time-dependent problems.
8We discuss the implementation of the spatially adaptive version of
9\c oomph-lib's unsteady Newton solver, \c
10Problem::unsteady_newton_solve(...), and explain why the assignment
11of initial conditions should be performed by overloading the
12\c Problem::set_initial_condition() function. We also
13discuss briefly how \c oomph-lib's generic dump and restart
14functions deal with adaptive meshes.
15
16For this purpose consider the following problem:
17
18

22<TD>
23<CENTER>
24<B>The two-dimensional unsteady heat equation with flux boundary
25conditions in a quarter circle domain</B>
26</CENTER>
27Solve
28\f[
29\sum_{i=1}^2\frac{\partial^2 u}{\partial x_i^2}
30= \frac{\partial u}{\partial t} + f\left(x_1,x_2,t\right),
31 \ \ \ \ \ \ \ \ \ \ (1)
32\f]
33in the quarter-circle domain \f$ D \f$, bounded by the coordinate axes
34and the unit circle, subject to Neumann boundary conditions,
35\f[
36\left. \frac{\partial u}{\partial n}\right|_{\partial D_{Neumann}}=
37- \left. \frac{\partial u}{\partial x_2}\right|_{\partial D_{Neumann}}=
38g_0,
39\ \ \ \ \ \ \ \ \ \ (2)
40\f]
41along the horizontal domain boundary \f$ \partial D_{Neumann} =
42\{ (x_1,x_2) | x_1 \in [0,1], x_2=0 \} \f$,
43and to Dirichlet boundary conditions,
44\f[
45\left. u\right|_{\partial D_{Dirichlet}}=h_0,
46\ \ \ \ \ \ \ \ \ \ (3)
47\f]
48elsewhere.
49\image html domain.gif "Sketch of the domain and the boundary conditions. "
50\image latex domain.eps "Sketch of the domain and the boundary conditions. " width=0.75\textwidth
51The initial conditions are given by
52\f[
53u(x_1,x_2,t=0)=k_0(x_1,x_2),
54\ \ \ \ \ \ \ \ \ \ (4)
55\f]
56where the functions \f$ f, g_0, \ h_0\f$ and \f$ k_0\f$ are given.
57</TD>
58</TR>
59</TABLE>
60</CENTER>
61
62We choose the functions \f$ f, g_0, \ h_0\f$ and \f$ k_0\f$
63so that
64\f[
65u_0(x_1,x_2,t) = \tanh\bigg[1-\alpha\bigg(\tan\Phi
66\big(x_1-\beta\tanh[ \gamma\cos\left(2\pi
67t\right)]\big)- x_2\bigg)\bigg]
68 \ \ \ \ \ \ \ \ \ \ (5)
69\f]
70is the exact solution.
71
72
73The solution represents the "usual" tanh profile, whose steepness
74is controlled by the parameter \f$ \alpha \f$ so that for \f$ \alpha
75\gg 1 \f$ the solution approaches a step. The step is oriented
76at an angle \f$ \Phi \f$ against the \f$ x_1-\f$ axis and its position varies
77periodically. The parameter \f$ \beta \f$ controls the amplitude
78of the step's lateral displacement, while \f$ \gamma \f$ determines the
79rate at which its position changes. For \f$ \gamma \gg 1 \f$ ,
80the step remains stationary for most of the period and then
81translates rapidly parallel to the \f$ x_1- \f$ axis, making this a very
82challenging problem.

110\section spatial_adapt Background: Spatial adaptivity in time-dependent problems
111
112Enabling spatial adaptivity in time-dependent problems involves
113essentially the same steps as for steady problems:
114- The domain must be discretised with a mesh that is derived
115 from the \c RefineableMesh base class.
116- An \c ErrorEstimator object must be created and passed to the
117 mesh.
118- The empty virtual functions \c Problem::actions_before_adapt() and
119 \c Problem::actions_after_adapt() may be overloaded to
120 perform any actions that are required before or after the mesh
121 adaptation, such as the deletion or recreation of any \c
122 FaceElements that apply flux boundary conditions.
123.
124Once these steps have been performed, a spatially adaptive
125solution can be computed with a three-argument version of \c oomph-lib's
126unsteady Newton solver \c Problem::unsteady_newton_solve(...):
127
128\code
129Problem::unsteady_newton_solve(dt,max_adapt,first)
130\endcode
131
132The arguments to this function are as follows:
133- The \c double \c dt specifies the (fixed) timestep.
134- The \c unsigned \c max_adapt specifies the maximum number of
135 spatial adaptations allowed.
136- The \c bool \c first indicates if the first timestep is performed.
137 This argument is required to allow the automatic re-assignment of
138 the initial conditions following any mesh adaptations during the
139 computation of the first timestep.
140.
141Given these arguments, the unsteady Newton solver solves the
142non-linear system of spatially and temporally discretised equations
143to advance the solution from time \f$ t \f$ to \f$ t + dt \f$ .
144Once the solution at time \f$ t + dt \f$ has been obtained,
145error estimates are computed for all elements. If any
146elemental error estimates are outside the target range, the solution
147is rejected and the mesh is adapted. In the course of mesh adaptation
148the existing solution (the nodal values \e and the history
149values) at time \f$ t \f$ are interpolated onto the new mesh
150 before recomputing the solution.
151This process is repeated until the error estimates are within
152the target range, or until the maximum number of adaptations, specified
153by the parameter \c max_adapt, is exceeded, just as in the steady case.
154
155Here is an illustration of the procedure for a 1D problem:
156
157\image html adapted.gif "Sketch of the mesh adaptation for time-dependent problems. "
158\image latex adapted.eps "Sketch of the mesh adaptation for time-dependent problems. " width=0.75\textwidth
159
160 This procedure is the obvious generalisation of the procedure for steady
161problems. However, in time-dependent problems two additional
162issues arise:
163-# In a steady problem the interpolation of the solution onto the
164 adapted mesh (step 4 in the above sketch) merely serves to
165 provide an initial guess for the solution on the refined
166 mesh. It is irrelevant if the interpolation from the coarse mesh
167 provides a poor approximation of the actual solution as
168 the solution is completely recomputed anyway.
169 \n\n
170 In an unsteady problem, we also have to interpolate the history
171 values (the solution at previous timesteps in a BDF scheme) onto
172 the adapted mesh. Their values are \e not
173 changed when the solution is advanced from time \f$ t \f$ to \f$ t+ dt. \f$
174 In time-dependent problems, the benefit of repeated mesh
175 adaptations (i.e. \c max_adapt > 1) is therefore
176 limited by the fact that mesh refinement cannot improve their
177 accuracy -- the history values are always given by
178 the (possibly poor) approximations obtained by interpolation
179 from the coarser mesh employed at the previous timestep. We
180 therefore recommend limiting the number of spatial adaptations to
181 \c max_adapt = 1. We stress that, in practice, this is not a serious
182 restriction because the time-integration procedure will only provide
183 (temporally) accurate results if the timestep \c dt is so small that the
184 solution at time \f$ t \f$ only differs slightly from that
185 at time \f$ t + dt \f$ . One level of mesh adaptation per timestep
186 should therefore be sufficient to adapt the mesh
187 in response to these changes.
188-# The only exception to this recommendation arises during the computation of
189 the first timestep, illustrated in the following sketch:
190\image html adapted_ic1.gif "Sketch of the mesh adaptation during the computation of the first timestep. "
191\image latex adapted_ic1.eps "Sketch of the mesh adaptation during the computation of the first timestep. " width=0.75\textwidth
192 When computing the first timestep, the solution on the initial mesh
193 will have been created by assigning the nodal values according to the
194 analytical initial condition
195 (4). If the initial mesh is very coarse
196 (as it should be), the finite-element representation of the
197 initial condition is likely to be very poor, as shown in the above
198 sketch. Clearly, the interpolation
199 from the coarse onto the fine mesh cannot recover any small-scale
200 features in the initial condition that were missed by its
201 representation on the coarse mesh. It is therefore better to
202 re-assign the initial condition (the values \e and the history values!)
203 on the adapted mesh, as shown in this sketch:
204\image html adapted_ic2.gif "Sketch of the modified mesh adaptation during the computation of the first timestep. "
205\image latex adapted_ic2.eps "Sketch of the modified mesh adaptation during the computation of the first timestep. " width=0.75\textwidth
206 With this procedure, repeated mesh adaptations will
207 improve the accuracy of the solution, therefore much larger
208 values of \c max_adapt can (and should!) be specified
209 when the first timestep is computed.
210 The unsteady Newton solver \c Problem::unsteady_newton_solve(...)
211 performs the revised procedure if the boolean argument
212 \c first is set \c true. In that case,
213 the values and history values on the adapted mesh are
214 (re-)assigned by calling the function
215 \code
216 Problem::set_initial_condition()
217 \endcode
218 which is defined as an
219 empty virtual function in the \c Problem base class. You should
220 overload it in your derived \c Problem to ensure that your specific
221 initial conditions are assigned
222 by the mesh adaptation procedures. [In fact, the function \c
223 Problem::set_initial_condition() is not quite empty -- not
224 re-setting the initial condition when performing mesh adaptations
225 during the first timestep of a time-dependent simulation seems "so
226 wrong" that the function issues a warning message. Although the
227 overloading of this function is not strictly necessary if the initial
228 conditions can be represented exactly by the interpolation from the
229 coarse mesh onto the fine mesh, we consider it good practice to
230 do so, for reasons discussed in <A HREF="../../../axisym_navier_stokes/spin_up/html/index.html#good_practice_ics">another tutorial</A>.]
231.

237\section overview Overview of the driver code
238
239Equipped with this background information, the driver code for our
240example problem is easy to understand, if somewhat
241lengthy. [Using an example with Dirichlet boundary conditions
242along the entire domain boundary would have shortened
243the code significantly but we deliberately chose an example with
244Neumann boundary conditions to demonstrate that the functions
245\c Problem::actions_before_adapt()
246and \c Problem::actions_after_adapt() may be used exactly as in
247the steady computations.] We will not discuss the
248methodology for applying flux-type boundary conditions in problems
249with spatial adaptivity in detail, but refer to the discussion provided
250in the <A HREF="../../../poisson/two_d_poisson_flux_bc_adapt/html/index.html">
251 earlier steady example.</A>
252
253Overall, the code is a straightforward combination of the driver code for
254the <A HREF="../../../poisson/two_d_poisson_flux_bc_adapt/html/index.html">
255steady Poisson problem with flux boundary conditions and spatial
256adaptivity</a> and the driver code for the
257<A HREF="../../two_d_unsteady_heat/html/index.html"> unsteady heat equation
258without spatial adaptivity. </A>

273\section ellipse Representing the curvilinear domain boundary by a GeomObject
274
275As discussed <A HREF="../../../poisson/fish_poisson2/html/index.html">
276elsewhere,</A> \c oomph-lib's mesh adaptation procedures require
277curvilinear domain boundaries to be represented by
278\c GeomObjects which describe the object's shape via their member
279function \c GeomObject::position(...). This function exists
280in two versions:
281- The two-argument version, \c GeomObject::position(xi,r)
282 computes the position vector, \c r, to the point on/in the \c GeomObject,
283 parametrised by the vector of intrinsic coordinates, \c xi.
284- The three-argument version \c GeomObject::position(t,xi,r), where
285 \c t is an \c unsigned, computes the position vector at
286 the \c t - th previous timestep.
287.
288In the current problem, the domain boundary

290function are identical. Here is the complete source code for

303\section main The main function
304
305As before, we use command line arguments to (optionally) specify
306a restart file. We store the command line arguments in
307the namespace \c CommandLineArgs and build the \c Problem object,
308passing the pointer to the source function. Next we specify the
309time-interval for the simulation and set the error targets for
310the spatial adaptation.
311
312\dontinclude two_d_unsteady_heat_adapt.cc
313\skipline start_of_main
314\until min_permitted_error
315
316We create and initialise the boolean flag that indicates if the first timestep
317is computed, and choose a large initial value for the number of permitted mesh
318adaptations. We then assign the initial conditions on the coarse
319initial mesh and retrieve the timestep (chosen when the
320initial conditions are assigned in \c set_initial_condition() )
321from the problem's \c Time object.
322
323\skipline Set IC
324\until ->dt
325
326If the simulation has been restarted, the first timestep is not the step
327at which the initial condition has to be assigned, therefore
328we reset the \c first and \c max_adapt parameters to their
329appropriate values. If the run is not restarted, the
330problem will have been built with a very coarse initial mesh
331(comprising just three elements). We don't need an error estimator
332to tell us that this is too coarse to represent the solution
333accurately and apply two levels of uniform refinement
334before solving the problem. Note that we refine the entire
335problem, not just the mesh to ensure that
336\c Problem::actions_before_adapt() and \c
337Problem::actions_after_adapt() are executed and the
338equation numbering scheme is re-generated. \c Problem::refine_uniformly()
339also interpolates the solution from the coarse initial mesh onto the
340refined mesh but, as discussed above, this will lead to a very poor
341representation of the initial condition. Therefore we re-assign
342the initial condition on the refined mesh and document the
343finite-element representation of the initial condition.
344
345\skipline If restart:
346\until doc_solution
347
348The time-stepping loop itself is very similar to that used in the
349<A HREF="../../two_d_unsteady_heat/html/index.html">
350example without spatial adaptivity</A>. Here we call the
351three-argument version of the unsteady Newton solver
352\c Problem::unsteady_newton_solve(...) and re-set the
353parameters \c max_adapt and \c first to their appropriate values
354once the first step has been performed.
355
356\dontinclude two_d_unsteady_heat_adapt.cc
357\skipline Find number of steps
358\until end of main

450\section restart Dump/restart with spatial adaptivity
451
452It is worth examining the
453dump and restart functions, however, as they demonstrate that
454the generic versions defined in the \c Problem base class
455can also deal with adaptive problems -- a non-trivial task!
456
457\skipline start_of_dump_it
458\until end of restart
459
460Since details of their implementation are hidden from the user, we
461briefly comment on the various tasks performed by these functions.
462The main task of the \c Problem::read(...) function is to
463read values (and history values) of all \c Data objects from a file and to
464assign these values to the appropriate \c Data (and \c Node) objects in the
465\c Problem. This assumes that the \c Problem's
466constituent \c Meshes, elements, \c Nodes and \c Data objects have been
467created, and that the \c Problem's various pointer-based lookup schemes
468access them in the order they were in when the \c Problem was dumped
469to the restart file. In a non-adaptive computation, the number of
470elements and the number of \c Data objects remain constant throughout the
471simulation and the \c Problem::read(...) function can be called
472as soon as the \c Problem has been built -- usually by its
473constructor. (The \c Problem constructor always builds and enumerates
474its constituent objects in the same order.)
475
476In a simulation with spatial adaptivity the number of elements, \c
477Nodes and \c Data objects varies throughout the computation.
478It is therefore necessary to re-generate the \c Problem's refinement pattern
479before the \c Data values can be read from the restart file. This is achieved
480(internally) by calling the function \c
481RefineableMesh::dump_refinement(...) for all refineable meshes
482before the \c Data is dumped. This function writes the \c Mesh's
483refinement pattern to the restart file, using a format that
484can be read by the corresponding member function
485\c RefineableMesh::refine(...) which adapts an unrefined mesh so that
486its topology and the order of its \c Nodes and elements is recreated.

493\section comments Comments and exercises
494
495The plots below show the time history of various parameters.
496- The upper graph compares the solution at the control node
497 (red line) against the exact solution (green line).
498- The middle graph shows the position of the step by plotting its
499 intercept with the \f$ x_1- \f$ axis as a function of time, and
500 the error of the solution.
501- The lower graph illustrates the evolution of the adaptive spatial
502 refinement process: The green line illustrates the total number of
503 elements; the blue and red lines show the number of elements that
504 are refined and unrefined at each timestep.
505
506\image html trace.gif "Time history of the solution. "
507\image latex trace.eps "Time history of the solution. " width=0.75\textwidth
508
509
510The plots illustrate clearly how the mesh is adapted as the step
511moves through the domain -- the peaks in the number of refined/unrefined
512elements per timestep coincide with the periods during which the step moves
513very rapidly. The increase in the error during these phases is mainly
514due to the temporal error -- the <A HREF="../figures/step_soln.avi">
515animation</A> shows that the computed solution lags behind
516the exact one. We will address this by adding adaptive time-stepping
517in <A HREF="../../two_d_unsteady_heat_2adapt/html/index.html">
518another example.</A>
