3\mainpage Demo problem: A one-dimensional eigenproblem
4In this document, we demonstrate how to solve a 1D eigenproblem
5(eigenvalues of the 1D Laplace operator in a bounded domain) by
6creating custom elements. The tutorial is similar in spirit to the
7<a href="../../../quick_guide/html/index.html"> quick guide </a> but far
8less detailed.

13<CENTER>
14<B>One-dimensional model eigenvalue problem</B>
15</CENTER>
16Solve
17\f[
18\frac{\partial^2u}{\partial x_1^2} + \lambda u = 0,
19 \ \ \ \ \ \ \ \ \ \ (1)
20\f]
21in the domain \f$D =\left\{ x_1 \in
22[0,1] \right\}\f$,
23with homogeneous Dirichlet boundary conditions
24\f[
25\left. u\right|_{\partial D}=0.
26\ \ \ \ \ \ \ \ \ \ (2)
27\f]
28</TD>
29</TR>
30</TABLE>
31</CENTER>
32
33 An eigenfunction must be non-zero by definition and so the exact
34solutions are given by the countably infinite set:
35\f[
36 u_{n} = \sin (\sqrt{\lambda}_{n} x_{1}), \ \ \ \ \ \ \
37 \lambda_{n} = n^{2}\pi^{2}.
38\f]
39
40We provide a detailed discussion of the driver code
41<A HREF="../../../../demo_drivers/eigenproblems/harmonic/harmonic.cc">
42harmonic.cc</A> which solves the problem for the first four eigenvalues.
43
44In \c oomph-lib all eigenproblems are solved by using interfaces to third-party
45libraries. The default is LAPACK's (direct) QZ solver which is included with
46the library. An interface is also provided to the ANASAZI solver, part
47of The Trilinos Project, a suite of iterative solvers.
48Thus, if you want to solve small problems, use the default QZ
49solver; for larger problems, we recommend using ANASAZI.

54\section theory Overview of the theory
55
56 A discrete (linear) eigenproblem takes the form
57\f[ J_{ij} V_{j} = \lambda M_{ij} V_{j}, \ \ \ \ \ \ \ \ (3) \f]
58 where \f$ V_{j} \f$ is the vector of discrete unknowns, \f$ \lambda \f$ is
59 the eigenvalue, and \f$ J_{ij} \f$ and \f$ M_{ij} \f$ are two matrices:
60 \f$ J_{ij} \f$ (often denoted \f$ K_{ij} \f$ in the
61 literature) is known as the stiffness matrix and \f$ M_{ij}
62 \f$ is termed the mass matrix. The third-party libraries mentioned
63 above all solve systems of the form (3), and within \c
64 oomph-lib we must simply provide the required mass and stiffness
65 matrices.
66
67\subsection linear_stab Linear Stability Theory
68
69In many cases, eigenproblems arise from
70linear stability analyses of steady solutions to our set of nonlinear
71residuals. For example, if we have a time-dependent system of equations
72\f[ \frac{\partial u}{\partial t} = \mathcal{F}(u), \f]
73then the residual formulation is
74\f[ \mathcal{R}(u) = \mathcal{F}(u) - \frac{\partial u}{\partial t} =
750; \ \ \ \ \ \ \ (4) \f]
76and steady solutions, \f$ u_{s} \f$ are found by setting \f$ \partial
77u / \partial t =
780 \f$. Note that in a nonlinear system there may be many possible
79steady solutions for the same set of parameter values.
80
81A linear stability analysis proceeds by assuming that the steady
82solution is perturbed by a small amount, so that
83\f$ u = u_{s} + \epsilon \hat{u} \f$, where \f$ |\epsilon| \ll
841\f$. If we substitute this expression into the governing equation
85(4) and
86neglect terms that involve \f$ \epsilon^{2} \f$ (on the assumption that they
87are really small) then we obtain the equation:
88\f[ \mathcal{F}(u_{s}) + \epsilon \left[\frac{\partial
89\mathcal{F}(u_{s})}{\partial u} \hat{u} - \frac{\partial
90\hat{u}}{\partial t} \right] \approx 0. \f]
91Note that we have used a Taylor expansion of \f$ \mathcal{F} \f$, so
92we are assuming that such an operation "makes sense". The solution \f$
93u_{s} \f$ is a steady state, so by definition \f$\mathcal{F}(u_{s}) =
940\f$ and then we are left with the \b linear equation
95 \f[ \frac{\partial \mathcal{F} (u_{s})}{\partial u} \hat{u} =
96 \mathcal{J}(u_{s}) \hat{u} = \frac{\partial
97 \hat{u}}{\partial t}. \ \ \ \ \ \ \ \ \ (5) \f]
98 The Jacobian \f$ \mathcal{J} \f$ is exactly the same as that required by the
99 Newton method when solving the steady set of equations, \f$
100 \mathcal{F}(u) = 0.\f$
101
102The general solution of equation (5) takes the form \f$
103\hat{u} = \mbox{e}^{\lambda t} v\f$ because the only function that is
104proportional to its derivative is the exponential. The necessary
105functional form of the solution means that equation (5)
106becomes
107\f[ \mathcal{J}(u_{s}) v = \lambda v, \f]
108which is an eigenproblem in which the mass "matrix" is the identity.
109Thus, in order to assess the stability of a solution to our nonlinear
110system we must solve an eigenproblem using the Jacobian matrix, which
111is why we used the notation \f$ J_{ij} \f$ rather than \f$ K_{ij} \f$
112in equation (3). The solution is said to be linearly stable if the real
113part of all eigenvalues is negative because then the perturbation
114decays as \f$ t \to \infty\f$, but it is linearly unstable if even a single
115eigenvalue has positive real part. Note that in even more general equations
116the mass "matrix" is not necessarily the identity.
117

119\subsection weak Weak formulation of the problem
120
121 If we wish to solve the problem (1) using finite elements,
122we must first derive the weak form, which follows from the standard
123procedure of multiplication by a test function and integration by
124parts
125\f[ \int_{D} \frac{\partial u}{\partial x_{1}} \frac{\partial
126\phi^{(test)}}{\partial x_{1}}\,\mbox{d} x_{1} =
127 \lambda \int_{D} u\, \phi^{(test)}\,\mbox{d} x_{1}. \f]
128Note that the boundary terms are neglected as a consequence of the
129Dirichlet boundary conditions.
130
131If we expand the \f$ u(x_{1}) = V_{j} \psi_{j}(x_{1}) \f$ in terms of
132known basis functions \f$ \psi_{j} \f$ and use
133the same basis functions as our test functions (Galerkin method), then the
134weak form becomes
135\f[ \int_{D} \frac{\partial \psi_{j}}{\partial x_{1}} \frac{\partial
136\psi_{i}}{\partial x_{1}}\,\mbox{d} x_{1} V_{j} = \lambda \int_{D}
137\psi_{i} \psi_{j}\,\mbox{d} x_{1} V_{j}. \f]
138On comparison with equation (3), we identify the Jacobian and
139mass matrices
140\f[ J_{ij} = \int_{D} \frac{\partial \psi_{j}}{\partial x_{1}} \frac{\partial
141\psi_{i}}{\partial x_{1}}\,\mbox{d} x_{1}, \ \ \ \ \
142 M_{ij} = \int_{D}
143\psi_{i} \psi_{j}\,\mbox{d} x_{1}. \ \ \ \ \ (6) \f]

148\section impl Implementation
149
150 In \c oomph-lib, eigenproblems are formulated on an
151element-by-element basis in a similar way to standard
152problems. Eigenproblems make use of the function
153\code
154 GeneralisedElement::get_jacobian_and_mass_matrix(
155 Vector<double> &residuals,
156 DenseMatrix<double> &jacobian,
157 DenseMatrix<double> &mass_matrix);
158\endcode
159 where the Jacobian is the matrix \f$J_{ij}\f$ and the mass matrix is
160 \f$M_{ij}\f$ in equation (3). The residuals need not be
161 returned when solving an eigenproblem, but the interface is chosen so
162 that linear stability of solutions to the nonlinear system can easily be
163 calculated without reformulating the problem.
164
165 Once again, to avoid reinitialisation in multi-physics problems, the helper
166 function
167\code
168 GeneralisedElement::fill_in_contribution_to_jacobian_and_mass_matrix(
169 Vector<double> &residuals,
170 DenseMatrix<double> &jacobian,
171 DenseMatrix<double> &mass_matrix);
172\endcode
173 is used and this will be overloaded in our custom elements below.

219\section main The driver code
220
221In order to solve the 1D eigenproblem using \c oomph-lib, we represent the
222mathematical problem defined by equations (1)
223and (2) in a specific \c Problem object, \c
224HarmonicProblem, which is templated by the element type and the
225eigensolver. We use \c QHarmonicElement<3>, a quadratic element and our
226three different eigensolvers, including timing statements to compare
227the three approaches. Note that if you do not have Trilinos installed
228then only the LAPACK_QZ solver will do anything.
229
230The problem class takes a single argument corresponding to the number
231of elements used to discretise the domain and contains a member
232function \c solve that takes an integer used for documentation purposes.
233

247\section problem The problem class
248The \c HarmonicProblem is derived from \c oomph-lib's generic
249\c Problem class and the specific element
250type and eigensolver are specified as template parameters to make it easy for
251the "user" to change either of these from the driver code.
252
253\dontinclude harmonic.cc
254\skipline start_of_problem_class
255\until public Problem
256
257The problem class has four member functions:
258- the constructor \c HarmonicProblem(...)
259- the destructor \c ~HarmonicProblem()
260- the function \c solve(...)
261- the function \c doc_solution(...)
262.
263
264 The destructor merely cleans up the memory by deleting the objects that are
265allocated in the constructor, so we shall not discuss it further
266here. The \c doc_solution function is also simple and writes the
267eigenfunction to a file.

272\section constructor The Problem constructor
273In the \c Problem constructor, we start by creating the eigensolver
274specified by the second template parameter.
275We then discretise the domain using \c oomph-lib's \c 1DMesh object.
276The arguments of this object's constructor are the number of elements
277(whose type is specified by the template parameter),
278and the domain length. Next, we pin the nodal values on the domain
279boundaries, which corresponds to applying the Dirichlet boundary conditions.
280Finally we call the generic \c
281Problem::assign_eqn_numbers() routine to set up the equation
282numbers.

291\section solve Solving the problem
292
293 The \c solve(...) function is where all the action happens
294 and takes a single unsigned integer argument which is used as a
295 label to distinguish the output from different eigensolvers.
296
297 The function \c Problem::solve_eigenproblem(...) plays an equivalent
298 role to \c Problem::newton_solve(...) in eigenproblems. Here,
299 additional storage must be allocated for the
300 eigenvalues (a vector of complex numbers)
301 and eigenvectors (a vector of double vectors). The vectors will be
302 resized internally depending on the number of eigenvalues
303 returned. The number is not always the same as the number of
304 eigenvalues requested because both parts of a
305 complex conjugate pair of eigenvalues are always returned.
306 In the \c solve(..) function we first
307 allocate the required storage, specify the desired number of
308 eigenvalues and then solve the eigenproblem:
309
310 \skipline start_of_solve
311 \until solve_eigenproblem(
312
313 The rest of the function post-processes the output from the
314 eigensolver. In order to ensure repeatability of the output for our
315 self-tests the eigenvalues are sorted on the size of their real
316 part. The eigenfunction associated with the second smallest
317 eigenvalue is normalised to have unit length and then output to a file.
318
319 In order to output the eigenfunction the values must be transferred
320 to the nodal values so that it can be interpolated. This is performed
321 by the function \c Problem::assign_eigenvector_to_dofs(...). Note
322 that this function overwrites the existing nodal data, so a backup
323 must be taken if it is important. This can be done using the function
324 \c Problem::store_current_dof_values() and the stored values can be
325 recovered via \c Problem::restore_dof_values().
326
327 Finally, the sorted eigenvalues are reported and also saved to a file.
